## TL;DR — топ-10 пріоритетів

1. **P0 видимість.** `site:boostershop.website` у Google повертає 0 результатів. Сайт фактично не індексується. Прямі пошуки бренду — у видачі конкуренти, Booster Shop не видно. TECH-005 sitemap fix зроблено, але indexation так і не пішла.
2. **P0 контент-bug.** Promotion Pack Vol.7 (`/product/One-Piece-Boosters-Promotion-Pack-V7`) — таблиця характеристик зміщена: "Мова: Sealed", "Виробник: Без зважування", "Рік випуску: Bandai" тощо. Виглядає як bug у Twig template або в attribute mapping.
3. **P0 ризик SEO/довіри.** На `/catalog/Pokemon/Pokemon-booster-box` опис категорії пише "Найкращий спосіб отримати **гарантовані рідкісні картки**. Наші бустер бокси Pokémon TCG **постачаються напряму з Японії**". Це прямо порушує і project rules ("не обіцяти гарантовані хіти"), і п.5.4 публічної оферти ("Продавець не заявляє, що всі товари постачаються напряму з Японії"). Юридично-репутаційний ризик.
4. **P0 контент.** Booster box Mega Symphonia і Munics Zero у meta description мають "Гарантовані рідкісні карти", у тілі — emoji-блок "🎯 Що ти гарантовано отримуєш з боксу" з конкретною rarity-структурою. Це claims, які треба переписати під формат "виробник зазвичай закладає розподіл рідкостей, але не гарантує конкретну карту".
5. **P1 H1 coverage.** Homepage не має H1 — найголовніший заголовок "Booster Shop — від колекціонера для колекціонерів" розмічений як H2 (`##`). Підтверджує пункт #2 з owner candidate tasks.
6. **P1 conversion blocker.** На більшості бустерів стоїть "Мінімальна кількість для замовлення: 2". Це або бізнес-правило, яке треба явно показувати на product card / на homepage (зараз ціна "150 грн" + кнопка "Купити" вводить в оману, бо кнопка приведе до помилки), або помилка налаштувань. Звертаю увагу: на boxes мінімум 1 — отже логіки немає чіткої.
7. **P1 product card states.** Поле "Наявність" показує то число ("Наявність: 51"), то статус ("Передзамовлення", "Закінчився"), і це присутнє в product card і product page. UX потребує єдиного state machine — підтверджує пункт #5 owner tasks.
8. **P1 SEO data integrity.** meta-keywords багатьох категорій і товарів містять "карточки покемон **оригинал**" (російський правопис), "**настільні ігри для дітей**" (нерелевантний keyword spam), а також невідповідності типу "Знижка від 10 шт" у meta description EB-03, тоді як tier-price реально від 5 шт.
9. **P1 індексаційна гігієна.** URL `/product/Pokémon-Trainer-Box-SVP` містить не-ASCII символ `é` у slug. Це не блокує Google повністю, але ускладнює sharing, ламає мої fetcher-перевірки і потребує redirect на ASCII-варіант.
10. **P1 структурна inconsistency.** Категорія One Piece не має підкатегорій (для Pokémon є: бустери, бокси, набори). Аналогічно: різна якість опису і шаблону характеристик у boxes (Hot Wind Arena/Super Electro Breaker мають повний шаблон, Mega Symphonia/Munics Zero — обрізаний emoji-варіант). Потрібен content normalization pass.

## Розділ A. Видимість і індексація — критичне

**Що показала перевірка:**
- WebSearch `site:boostershop.website` — нуль результатів від цього домену.
- WebSearch "Booster Shop японські бустери покемон Україна" — у TOP-10 видачі: dragonshop.com.ua, prom.ua, pokedow.com.ua, geekach.com.ua, epicentrk.ua, joom.com, booster-ua.com.ua, bi.ua, yestoys.com.ua. Booster Shop не присутній.
- WebSearch "boostershop.website Pokemon TCG" — нуль попадань.

**Інтерпретація.**
Це підтверджує, що відомий блокер TECH-005 (Brotli sitemap response) усунуто на стороні сервера, але Google ще не переіндексував сайт або відсутні достатньо сигналів. У контексті того, що сайт містить нормальні title, description, canonical, og — це не контентна проблема. Проблема операційна: треба переконатися, що GSC бачить sitemap у нормальному стані і ручно надіслати sitemap submission, після чого моніторити Coverage report.

**Що з цим робити (детальніше — розділ L, задача SEO-001):**
- ручне повторне submission sitemap у GSC;
- перевірити URL Inspection на 3–5 ключових URL (`/`, `/catalog/Pokemon`, `/catalog/One-Piece`, `/product/Pokemon-boosters-Mega-Symphonia`, `/information/oplata-i-dostavka`);
- зафіксувати у roadmap дату наступної перевірки (через 7 і 14 днів);
- не чіпати root sitemap.xml, robots.txt, redirects, canonical до моменту коли GSC покаже indexed.

**LLM-видимість.**
WebSearch і знання LLM про "Booster Shop" як TCG-магазин в Україні — нуль. Це нормально для нового сайту, але прискорюється:
- зовнішніми згадками (OLX user page вже є — посилання в footer), сторонніми блогами або форумами TCG-комʼюніті, дописами в Telegram-каналах;
- наявністю структурованих даних Organization і Product;
- стабільним indexed-станом у Google.

## Розділ B. Технічний SEO (що видно з HTML)

**Що добре:**
- Canonical-теги на всіх перевірених сторінках, self-canonical, без помилкових редиректів.
- og:title, og:description, og:image, og:type, twitter card — присутні всюди.
- meta-google-site-verification присутній на всіх сторінках.
- meta-robots `noindex,follow` встановлений на `/account/login` і `/account/register` — правильно.
- meta-viewport `width=device-width, initial-scale=1` всюди — норм.
- charset utf-8, content-type правильний.
- Title-патерни наповнені і унікальні per page.

**Що погано / неясно:**
- meta-keywords використовується. Це не SEO-фактор у Google років 15, але і не шкодить — крім випадків, де там керівник російською ("оригинал" замість "оригінал") або keyword stuffing ("настільні ігри для дітей" у Pokémon-категоріях).
- meta-description на /special відсутній взагалі. Title теж порожній за змістом ("Акції").
- Чи рендериться JSON-LD у HTML — fetch повертає тільки видимий markdown без `<script type="application/ld+json">`. Потрібна окрема перевірка через Rich Results Test / view-source (див. data gaps).
- HTTP-headers, security headers, Content-Encoding, Cache-Control — не доступні в fetch output. Згідно roadmap TECH-001 і historical context, сайт раніше віддавав `X-Powered-By: PHP/8.0.30`. Не міняти без окремого smoke-test.
- hreflang відсутній — це нормально для UA-only сайту, не виправляти.

**Окрема технічна знахідка:**
- Логотип у header використовує URL `https://boostershop.website/image/catalog/One Piece/BS Big logo.png` — файл лежить у папці "One Piece", з пробілами в назві. Це не SEO-проблема як така, але:
  - брендингово невірно (брендовий ассет у категорії конкретного TCG);
  - пробіли в URL — ризик encoding edge cases;
  - дублюється у og:image на кількох сторінках.

## Розділ C. Головна сторінка

**Url:** `https://boostershop.website/`

**Знахідки:**
- Title і meta description зрозумілі і релевантні: "One Piece TCG та Pokémon карти | Японські бустери в наявності".
- **Немає H1.** Найвищий heading на сторінці — H2 "Booster Shop — від колекціонера для колекціонерів". Це SEO-помилка і UX-помилка водночас.
- Featured products: 4 товари. Маленька кількість, але адекватна для shop з 14 in_stock SKU.
- Опис магазину текстовий блок є — короткий, у тоні бренду, без води.
- Footer повний: інформація / додатково / покупцю / контакти. ФОП Леусенко, телефон, Telegram, email. OK.
- Шильд "Бустер Шоп © 2026" — підозра на hardcoded year (треба перевірити Twig: чи виводиться через `{{ "now"|date('Y') }}`).
- Cookie banner присутній. Лінк "Докладніше" веде на `/publichna-oferta?route=information/information.info` — це dirty URL з зайвим параметром (sub-route info на information page). Не критично, але непрофесійно.

**Запропоновані зміни (детальніше у розділі L):**
- додати H1 (наприклад "Оригінальні бустери Pokémon TCG та One Piece Card Game" або відверто бренд "Booster Shop — японські бустери Pokémon і One Piece");
- розглянути додавання H2 для секції featured ("Рекомендовані товари" зараз H3 `###`);
- додати 2–3 FAQ у нижній частині home для AI/search relevance (підтверджує owner task #4);
- винести "Бустер Шоп © 2026" у динамічний рік.

## Розділ E. Товарні сторінки

Перевірено 14 з 15 (Pokémon Trainer Box SVP — fetch заблокований через é у URL — окремий знахідка про ASCII slug).

### E.1 Шаблонні проблеми (стосуються майже всіх товарів)

**Мінімальна кількість замовлення = 2** на більшості бустерів (всі бустери Pokémon і One Piece, окрім Mystery Mix і booster boxes де min = 1). Якщо це бізнес-правило: треба явно показувати "Продається мінімум від 2 шт" на product card на категоріях. Зараз кнопка "Купити" з ціною "150 грн" вводить в оману.

**Поле "Наявність":**
- Mega Symphonia: "Наявність: 51"
- EB-03: "Наявність: 23"
- Munikis Zero: "Наявність: Передзамовлення"
- Mix low pull: "Наявність: Закінчився"
- Mega Symphonia BB: "Наявність: Закінчився"
- OP-10: "Наявність: Передзамовлення"
- OP-12: "Наявність: Передзамовлення"
- OP-14: "Наявність: Закінчився"

Це state machine, але до користувача воно доходить як вільний текст. На product card теж треба chip/badge: "В наявності", "Передзамовлення", "Закінчився". Прямо підтверджує owner task #5.

**Tier-pricing і Discount:**
- Більшість бустерів: 5 або більше = -10 грн (Pokemon) / -10 грн (One Piece).
- OP-12 — tier від 10, не від 5. Inconsistency.
- Mix low pull — від 15 шт = 65 грн при базі 80 грн (тобто -15 грн при mass purchase). Тут логіка нормальна (це low-pull pack).

**Опис і FAQ:**
- На бустерах є якісний шаблон з блоками: Опис → Чому саме → Chase Cards → Чому купують → FAQ. Дуже добре написано (особливо Mega Symphonia, EB-03, OP-07, OP-10, OP-11, OP-12, OP-14, Ninja Spinner).
- На bосах Hot Wind Arena і Super Electro Breaker — шаблон такий самий, добре.
- На boxes Mega Symphonia і Munics Zero — обрізаний шаблон (без Chase Cards, без полного "Чому саме"), з emoji-блоком "🎯 Що ти гарантовано отримуєш з боксу" і списком SR/SAR/UR/AR/RR. Це шаблонна inconsistency + потенційно ризикові claims.

**Характеристики:**
- На бустерах: блок з 9 полів у правильному порядку (Назва сету, Мова, Тип, Кількість, Зважування, Стан, Походження, Виробник, Рік випуску). OK.
- На boxes Mega Symphonia: тільки 6 полів, поле "Мова" = "Японія" (мова не Японія, а Японська/Japanese). Рік випуску = "01.08.2025" (формат дата на одному, "23.01.2026" на Munics Zero, "2024" на Super Electro Breaker, "2025" на Hot Wind Arena). Date-format inconsistent.
- На Promotion Pack Vol.7: **повністю зламана таблиця, поля зміщені**.

**Теги:**
- 10–15 тегів на товар. Більшість релевантні. Але деякі дублюються latin/cyrillic ("Pokemon" і "Pokémon TCG" як окремі теги). Може створювати thin tag pages.

**Відгуки:**
- "0 відгуків / Написати відгук" на всіх товарах. Реальна історія відгуків знаходиться на OLX (footer link), і це чесна політика — не вигадувати reviews. Можна або сховати тег "0 відгуків", або зробити inline пояснення з посиланням на OLX.

**Огорожа продукту по типу:**
- Sealed booster Pokémon: 5 карт.
- Sealed booster One Piece: 6 карт (для OP-07, OP-10, OP-11, OP-12, OP-14, EB-03).
- Booster box: 30 бустерів х 5 карт.
- Promotion Pack Vol.7: 5 фіксованих карт (правильно прописано як not random).
- Mystery Mix: 5 sealed бустерів + 1 holo card.
- Low pull mix: змінний.

## Розділ P. Нові P0 знахідки (підтверджені скрінами)

### P.1 Акції зламались — підтверджено
Власник дав 2 скріни:
- `/catalog/Pokemon` — MIX low pull показує `₴150.00` як єдину ціну, без перекреслення і без акційної. Той самий товар, у якого в admin → "Знижка" tab стоїть 80 (qty 1) і 70 (qty 15), Акція=ON.
- `/special` — "Немає товарів по акції" і кнопка "Продовжити".

Це не Special vs Discount mapping. Це **повне зникнення активних знижок** — ні tier discount mix lowpull (80 → 150 зараз), ні box special (3200 → 3000) не рендериться у фронті.

**Гіпотези по приоритету:**

1. **LiteSpeed Cache** хешує product object без special_price після того, як cron спрацював `update product set special_price = NULL` (або подібне). Потенційно — auto-task в OpenCart який скидає expired specials, але стрільнув по всіх. Перевірити: admin → Catalog → Products → Special prices або через БД SELECT.
2. **OpenCart extension оновлення** (наприклад тема, payment, або тhe extension SimpleCheckout) перетягнуло product.twig або product_list.twig і пропустив render блоку `special`. Перевірити: `view/template/product/product.twig` і `product/category.twig` — чи є блок `{% if special %}`.
3. **Cache plugin зливаючи tier discount** — в OpenCart 4 є відмінність між `product_special` і `product_discount`. ### P.2 Sitemap-gsc.xml віддається як binary — reproducible зі мого боку
Контекст: при fetch через mcp web_fetch на `https://boostershop.website/sitemap-gsc.xml` відповідь приходить як `[binary data]` під `Content-Type: application/xml`. Це 1:1 збігається з логом Google `66.249.73.* GET /sitemap-gsc.xml → 200 249 bytes` при реальному `Content-Length: 5006`.

Гіпотези по приоритету:
1. **LiteSpeed Cache + Brotli on-demand**. LiteSpeed-нативна Brotli compression живе окремо від Apache `mod_deflate`, працює як LSWS plugin, і її поведінка не повністю контролюється `.htaccess SetEnvIf`. На `.xml/.txt` може видаватись compressed body без HTTP header `Content-Encoding: br`, якщо LSWS думає що клієнт це підтримує (Accept-Encoding містить br).
2. **HTTP/3 / QUIC** — Google може пробувати QUIC. На QUIC compression negotiation відрізняється. LiteSpeed має admin → Server → QUIC. Тимчасово вимкнути `alt-svc` або QUIC і дочекатись нової fetch.
3. **OpenCart sitemap module ob_gzhandler** — якщо `sitemap-gsc.xml` генерується PHP (не static file), і модуль робить `ob_start("ob_gzhandler")`, тоді LiteSpeed подвійно компресує. Розкрити `system/library/seo/...` або `extension/...sitemap...` і подивитись.

**Handoff** (нова задача):

**TECH-005-DEEP** [P0, Technical SEO, Owner: Codex investigation, Status: Planned]
- Goal: знайти і усунути причину, чому Google отримує binary/compressed body для `/sitemap-gsc.xml` без правильного Content-Encoding header.
- Steps:
  1. Перевірити LiteSpeed Web Admin → Configuration → Server → Tuning → Compression → `Static gzip compression` і `Brotli`. Виставити: compression OFF для `application/xml` і `text/xml`.
  2. Перевірити LSCache Plugin settings (якщо встановлено LSCWP для OpenCart). У "Page Optimization → Optimization" або "Browser Cache → Cache TTL for sitemap" — переконатись що `.xml` не береться в кеш.
  3. Перевірити `.htaccess` на конфліктуючі блоки `mod_deflate` / `mod_filter` / `AddOutputFilterByType` для `application/xml`.
  4. Якщо нічого з вище — створити статичний sitemap файл (без PHP) і перевірити що його доставка нормальна. Якщо так — проблема в OpenCart sitemap generator.
  5. Підготувати curl test matrix: HTTP/1.1 + HTTP/2 + HTTP/3, з і без `Accept-Encoding`, з і без UA Googlebot.
- Do not touch: robots.txt, redirects, canonical, інші sitemap файли. Не міняти URL sitemap.
- AC: `curl -H "Accept-Encoding: gzip, deflate, br" https://boostershop.website/sitemap-gsc.xml -o /tmp/s.xml` повертає readable XML без binary content. У логах LiteSpeed бачимо реальний 5006 bytes для Googlebot.
- Rollback: backup `.htaccess` і LiteSpeed config перед будь-якою зміною.
- Smoke-test після фіксу: дочекатись 66.249.* fetch у логах і перевірити що response > 5000 bytes.
