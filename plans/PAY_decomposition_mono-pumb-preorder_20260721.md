## 1. Висновок

Три окремі задачі замість однієї розмитої. **(А)** PAY-001 mono — sandbox повністю пройдено (4 телефони, регресія чиста), лишились Phase 2 UI + production-реєстрація. **(Б)** ПУМБ — нова задача PAY-002: API справді структурно ~90% схоже на mono (HMAC-SHA256/Base64, create→callback→confirm-на-відправці, ідемпотентність по store_order_id, return/cancel), тому робимо mono до кінця і клонуємо по шаблону; договір з ПУМБ треба запускати ВЖЕ (тестовий контур доступний лише після підписання — це найдовший ланцюжок). **(В)** «Передзамовлення» — НЕ кредитна механіка: власник хоче лише новий статус замовлення в адмінці. Виносимо в окрему маленьку задачу ORD-PREORDER, скоуп PAY-001 Phase 2 НЕ змінюється. Помилкове трактування «пре-замовлення × ПЧ» з progress-звіту Codex (розділ 6) — зняте власником 2026-07-21.

## 6. Codex handoff — технічний контекст

### 6.1 ПУМБ API проти mono — що збігається (база для клонування)

Підпис `Base64(HMAC-SHA256(body, secret))`; заголовки-пара store-id/signature (у ПУМБ — `X-Store-Id`/`X-Signature`); create → клієнтське підтвердження → callback + polling fallback; **confirm видачі = момент руху грошей** (як у mono — списання першого платежу), викликається вручну при реальній відправці; cancel до confirm / return після; ідемпотентність по `store_order_id` (ПУМБ на дубль явно віддає `409`); суми в копійках; products-масив у заявці. Архітектура mono_chast (transaction-таблиця, стани, callback-верифікація, poll) переноситься майже 1:1.

### 6.2 Ключові відмінності (таблиця для майбутнього PAY-002 патча)

| Параметр | mono ПЧ | ПУМБ |
|---|---|---|
| Флоу клієнта | підтвердження в застосунку mono по телефону | **redirect на `redirect_url`** банку + `return_url` назад |
| Платежі | 3/4/5 (правило власника, `available_parts_count`) | ПЧ: 3/4/6/9; окремо CREDIT до 24 міс (`program_type: PP\|CREDIT`) |
| Мін. сума | 500 грн (правило власника) | **1000 грн** (вимога банку) |
| Макс. сума | — | 100 000 грн ПЧ / 300 000 грн кредит |
| TTL confirm | невідомо (не блокує: НП-відправка = дні) | **30 календарних днів** у WAITING_FOR_CONFIRM, далі автоскасування |
| Статуси | 6 станів mono | 10 станів — мапити на ті самі 6 OC-статусів + окремо CLIENT_PROCESSING/CANCELED |
| Callback | підпис у заголовку | підпис у тілі; **відповісти 200 за 10 с**, 5 ретраїв по 15 хв; IP-whitelist |
| Sandbox | публічний, без договору | **лише після договору**; тест-телефони +38000000000{1,2,3} |
| Rate limit | є (докс mono) | 100 req/хв (`429`) |
| Комісії | 3→2.9%, 4→4.1%, 5→5.9% (підтверджено) | **невідомі — взяти у менеджера, не вигадувати** |

### 6.3 Що це означає для коду

- Модалка вибору провайдера (вже в дизайні PAY-001-UI) стає єдиною точкою входу для обох: картка mono (активна після Phase 2) + картка ПУМБ (активується в PAY-002). Фільтр доступності провайдера по сумі кошика: mono ≥500, ПУМБ ≥1000 і ≤ліміт.
- ПУМБ — окремий extension (`extension/pumb_credit` або аналог) зі своєю transaction-таблицею і своїм `payment_pumb_*_status`-прапорцем; НЕ розширювати mono_chast всередині.
- Callback URL: цитата фахівця ПУМБ — «потрібна інфо з розділу кредити, вкладка пумб, callback url і все». Callback-роут з'явиться з extension-скелетом PAY-002; URL можна зарезервувати і повідомити ПУМБ заздалегідь (формат: `?route=extension/pumb_credit/payment/pumb_credit.callback` — фінальний вигляд у handoff PAY-002).
- NCRM: окрема майбутня міграція `credit_pumb_*` payment types (за зразком 0013), лише коли відомі фінальні набори платежів і комісії.

### 6.4 ORD-PREORDER (окремо, поза PAY)

Скоуп: додати order status «Передзамовлення» в OpenCart адмінку (`oc_order_status`, uk-локаль) + узгодити мапінг у NCRM order-sync (щоб новий статус не впав у «невідомий»). Це НЕ кредитна логіка і НЕ чекаут — ризик низький, але order-sync уже має статусні мапінги, тому патч має пройти перевірку проти `ncrm/supabase/functions/order-sync/index.ts`. TTL-питання з progress-звіту Codex знімається як блокер: для ПУМБ відповідь у доксах (30 днів), для mono — цікаво лише перед production, не для передзамовлень.

- ORD-PREORDER: створити тестове замовлення → виставити статус → перевірити відображення в адмінці та коректний прохід через order-sync у NCRM.

## 8. Ризики

- **Checkout/payment — HIGH-RISK зона:** кожен патч PAY-серії торкається живого флоу оплати; правила з RESET-handoff лишаються (status-прапорці, isolation, координація з ST-2c cutover, smoke test обов'язковий).
- **Комісії ПУМБ невідомі** — без тарифної сітки від менеджера не фіксувати нічого в NCRM і не обіцяти умов у UI.
- **Тестовий контур ПУМБ лише після договору** — якщо не запустити папери зараз, PAY-002 впреться в очікування на тижні.
- **Callback-вимоги ПУМБ жорсткіші** (200 за 10 с, IP-whitelist): якщо хостинг повільний на cold-start — ризик пропущених статусів; poll-fallback обов'язковий як у mono.
- **ORD-PREORDER зачіпає order-sync мапінги** — новий статус без мапінгу може зламати синк замовлень у NCRM; патч тільки разом з перевіркою index.ts.
- «90% схожості» — оцінка ПУМБ, не наша: не копіювати код сліпо, кожну відмінність з таблиці 6.2 проганяти явно.
