## 6. Codex handoff — технічний контекст

### 6.1 ПУМБ API проти mono — що збігається (база для клонування)

Підпис `Base64(HMAC-SHA256(body, secret))`; заголовки-пара store-id/signature (у ПУМБ — `X-Store-Id`/`X-Signature`); create → клієнтське підтвердження → callback + polling fallback; **confirm видачі = момент руху грошей** (як у mono — списання першого платежу), викликається вручну при реальній відправці; cancel до confirm / return після; ідемпотентність по `store_order_id` (ПУМБ на дубль явно віддає `409`); суми в копійках; products-масив у заявці. Архітектура mono_chast (transaction-таблиця, стани, callback-верифікація, poll) переноситься майже 1:1.

### 6.3 Що це означає для коду

- Модалка вибору провайдера (вже в дизайні PAY-001-UI) стає єдиною точкою входу для обох: картка mono (активна після Phase 2) + картка ПУМБ (активується в PAY-002). Фільтр доступності провайдера по сумі кошика: mono ≥500, ПУМБ ≥1000 і ≤ліміт.
- ПУМБ — окремий extension (`extension/pumb_credit` або аналог) зі своєю transaction-таблицею і своїм `payment_pumb_*_status`-прапорцем; НЕ розширювати mono_chast всередині.
- Callback URL: цитата фахівця ПУМБ — «потрібна інфо з розділу кредити, вкладка пумб, callback url і все». Callback-роут з'явиться з extension-скелетом PAY-002; URL можна зарезервувати і повідомити ПУМБ заздалегідь (формат: `?route=extension/pumb_credit/payment/pumb_credit.callback` — фінальний вигляд у handoff PAY-002).
- NCRM: окрема майбутня міграція `credit_pumb_*` payment types (за зразком 0013), лише коли відомі фінальні набори платежів і комісії.

### 6.4 ORD-PREORDER (окремо, поза PAY)

Скоуп: додати order status «Передзамовлення» в OpenCart адмінку (`oc_order_status`, uk-локаль) + узгодити мапінг у NCRM order-sync (щоб новий статус не впав у «невідомий»). Це НЕ кредитна логіка і НЕ чекаут — ризик низький, але order-sync уже має статусні мапінги, тому патч має пройти перевірку проти `ncrm/supabase/functions/order-sync/index.ts`. TTL-питання з progress-звіту Codex знімається як блокер: для ПУМБ відповідь у доксах (30 днів), для mono — цікаво лише перед production, не для передзамовлень.

### 6.5 Факти з договору ПУМБ (№ SF1/21.3.2/4510, 2026-07-21), що впливають на реалізацію і процеси

- **Оформлення кредиту — клієнт самостійно в мобільному додатку PUMB Online** (п.1.1, 1.3); магазин лише передає контактні дані покупця через API. Узгодити з redirect_url-флоу з API-доки при отриманні «Порядку обміну» — можливо, redirect веде саме в app-флоу.
- **Гроші:** банк перераховує кошти наступного робочого дня після передачі товару покупцю (свята/вихідні — до 2 роб. днів), комісію утримує сам із суми перерахування (п.2.1.3, 2.2.9). Для NCRM: сума надходження = ціна мінус комісія, окремий payment_type `credit_pumb_3/4/5`.
- **Рішення банку по заявці — не пізніше наступного робочого дня**; на цей час магазин зобов'язаний **резервувати товар** (п.2.2.2). Стик з логікою резервів/стоку — врахувати в handoff PAY-002.
- **Видача товару — в день отримання рішення** банку + повідомити банк через API про передачу (п.2.2.3) — це аналог confirm у mono.
- **Обраний флоу відвантаження (власник, 2026-07-21, уточнено — «комбінований» на базі варіанту 1):** сигнал «відвантаження товару» йде банку **в момент фізичної передачі посилки поштовому оператору** (завантаження у поштомат/відділення, створена ТТН), не в момент оформлення. Це вкладається в TTL 7 днів і трохи звужує вікно невикупу. Заявка переходить у `FOUNDED` (статусу нема в PDF v2.4 — чекати актуальний «Порядок обміну»), гроші на рахунку за 5–10 хв.
- **Автоматизація в OpenCart (проєктне рішення для PAY-002):** обидва напрями автоматичні. Вхідний: callback банку (клієнт підтвердив у застосунку, схвалення тощо) → модуль сам оновлює статус замовлення в адмінці — власник «бачить це» без ручних дій. Вихідний: власник міняє статус замовлення на «Відправлено» → event-hook на зміну статусу сам викликає API «відвантаження товару». Вимоги: ідемпотентність через transaction-таблицю (без подвійного confirm при повторній зміні статусу); якщо виклик API впав — явна помилка/алерт власнику, НЕ мовчазний фейл; сумісність з існуючим NCRM order-sync хуком на зміни замовлень (не конфліктувати). Той самий hook-патерн придатний і для mono confirm — уніфікувати в Phase 2/PAY-002.
- **Перший платіж клієнта у ПУМБ — через місяць після видачі** (слова власника з розмови з банком), не в момент confirm (відмінність від mono, де confirm одразу списує перший платіж). Наслідок для невикупу: клієнт на момент повернення ще нічого не платив → банку повертається повна сума; операційно це «майже звичайне повернення» + API-статуси банку (return по API, свій чек повернення в Checkbox — як зазвичай).
- Фіскалізація: власник фіскалізує чек у день замовлення (Checkbox), як і зараз — менеджер банку на це питання прямо не відповів; на код не впливає, у разі сумнівів власник уточнює у свого бухгалтера.
- **Повернення:** за заявою покупця — перерахувати банку вартість товару (мінус перший внесок клієнта) протягом **3 робочих днів** (п.2.2.4).
- **Рахунок-фактура з підписом і печаткою** покупцю на товар у кредит (п.2.2.2) — операційний процес власника, не код.
- **Реклама:** будь-які рекламні матеріали про купівлю в кредит ПУМБ **погоджуються з банком до публікації** (п.2.2.8) — стосується і UI (бейджі/модалка/банери з брендом ПУМБ). Макети надсилати на погодження до релізу PAY-002 UI.
- **Конфіденційність (п.2.2.7):** зміст договору і креденшли не розголошуються. **PDF договору в git-репо НЕ комітити**; у репо тримаємо лише мінімально потрібні робочі факти (цей розділ).
- Строк дії: до 31.12.2026, автопролонгація. Email банку: pumb.partnerhelp@fuib.com, документообіг — «Вчасно».

## 8. Ризики

- **Checkout/payment — HIGH-RISK зона:** кожен патч PAY-серії торкається живого флоу оплати; правила з RESET-handoff лишаються (status-прапорці, isolation, координація з ST-2c cutover, smoke test обов'язковий).
- **PDF v2.4 ≠ реальне API** (статус `FOUNDED`, TTL 7 днів замість 30 вже розійшлись з доками) — PAY-002 handoff писати ТІЛЬКИ від актуального «Порядку обміну» від банку, PDF v2.4 вважати орієнтиром, не специфікацією.
- **Варіант 1 (гроші в день відправки) переносить ризик невикупу на нас:** клієнт не забрав посилку → повернення через API + перерахування коштів банку за 3 роб. дні (п.2.2.4, мінус перший внесок клієнта). Потрібен операційний процес трекінгу невикуплених ПЧ-посилок (НП статуси) — інакше касовий розрив.
- **Реклама без погодження макетів = договірний ризик** (п.2.2.8, банк може перекласти штрафи на продавця) — жодного публічного UI/банера з брендом ПУМБ до письмового погодження макетів. Робоче тлумачення: погоджується шаблон, не кожна товарна сторінка — але це підтвердити письмово (розділ 5.2в).
- **Резервування товару + видача в день рішення банку** — операційні зобов'язання договору; якщо процес власника їх не витягує (наприклад, товар закінчився за час рішення), потрібен cancel-флоу в модулі.
- **Callback-вимоги ПУМБ жорсткіші** (200 за 10 с, IP-whitelist): якщо хостинг повільний на cold-start — ризик пропущених статусів; poll-fallback обов'язковий як у mono.
- **ORD-PREORDER зачіпає order-sync мапінги** — новий статус без мапінгу може зламати синк замовлень у NCRM; патч тільки разом з перевіркою index.ts.
- «90% схожості» — оцінка ПУМБ, не наша: не копіювати код сліпо, кожну відмінність з таблиці 6.2 проганяти явно.
