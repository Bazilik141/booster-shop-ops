## 0. Конвенції (для всіх таблиць)

- **PK:** сурогатний `id uuid DEFAULT gen_random_uuid()`, окрім явно довідкових. Натуральні ключі (sku) — `UNIQUE`, не PK, щоб їх можна було міняти без поломки зв'язків.
- **Гроші:** `numeric(12,2)` у грн (UAH) як базова; де є інша валюта вводу — зберігаємо четвірку `*_amount` + `*_currency` + `*_rate` + `*_uah`.
- **Заморозка конвертації (важливо):** `_rate` і `_uah` обчислюються **один раз на момент вводу** і зберігаються назавжди. Оновлення курсів у `currency_rates` **НЕ перераховує** вже внесені суми. Тобто внесені умовні $10 доставки залишаться тими ж грн і завтра після зміни курсу.
- **Дати/час:** `timestamptz`, `created_at DEFAULT now()`, `updated_at` (тригер).
- **М'яке видалення:** `is_active boolean` / `archived_at` де доречно (нічого фізично не видаляємо).
- **Назви:** snake_case, англійською; українські значення статусів — у `text` + довідник.
- **Аудит:** `created_at`, `updated_at`; для фінансово критичних — окремий лог змін (Stage 4).

## Порядок стадій

1. **Stage 1 (цей файл):** Ядро та входи собівартості — `app_config`, `currency_rates`, `products`, `product_prices`, `purchases`, `purchase_lots`.

# STAGE 1 — Ядро та входи собівартості

## 1.1 `app_config` — параметри (заміна аркуша `Налаштування`)

Типізований key-value для всіх коефіцієнтів. Прибирає «магічні числа» з коду.

| key | приклад значення | сенс |
|---|---|---|
| `credit_rate_monthly` | 0.03 | кредитна ставка/міс |
| `credit_months` | 2 | місяців кредитного обслуговування |
| `credit_servicing_pct` | = rate×months = 0.06 | надбавка в управлінську собівартість |
| `expected_income_haircut` | 1.05 | дисконт **очікуваного** прибутку (не фактичного) |
| `acquiring_pct` | 0.015 | еквайринг |
| `fop_control_pct` / `fop_control_min` | 0.005 / 0 | контроль оплати ФОП |
| `payback_fiz_pct` / `payback_fiz_fix` | 0.02 / 20 | післяплата фіз |
| `cost_start_date` | 2026-04-01 | дата старту складського обліку |
| `nbu_rate_buffer_pct` | 0.01 | +1% до курсу НБУ → курс для CRM |

Структура: `key text PK, value_num numeric, value_text text, value_date date, unit text, description text, effective_from date, is_active bool`.
- `[D1]` ✅ Зберігати `credit_rate_monthly` + `credit_months` як вхідні, а `credit_servicing_pct` — **обчислюване** (view), щоб не розсинхронити. (Зараз так і є: `B6=B4*B5`.)
- `[D2]` ✅ Параметри **effective-dated** (`effective_from`) — щоб зміна ставки не переписувала історію заднім числом.

## 1.2 `currency_rates` — курси (заміна `Курси`, + вимога §13.1)

| поле | тип | нотатка |
|---|---|---|
| `id` | uuid PK | |
| `currency` | text | `UAH`/`JPY`/`USD`/`EUR` |
| `rate_to_uah` | numeric(12,6) | скільки грн за 1 одиницю валюти |
| `as_of` | date | дата дії курсу |
| `source` | text | per-currency: `ПриватБанк`/`Monobank` API (USD/EUR, якщо доступно) → інакше `НБУ`; JPY=`НБУ`; `ручне` |
| `note` | text | |

- `UNIQUE(currency, as_of)`; `UAH` = 1.
- `[D3b]` ✅ **Щоденне (або раз на 2 дні) оновлення** додає **новий рядок** на дату (історія), не перезаписує старий. При вводі закупки береться курс із найсвіжішою `as_of ≤ дата операції`, множиться на `_amount` → `_uah` заморожується (див. §0).
- `[D3c]` ✅ Джерело курсу (пріоритет): відкрите API **ПриватБанку або Monobank** для USD/EUR, якщо доступне; інакше **НБУ**. **JPY — завжди НБУ.** Коли джерело НБУ → **курс для CRM = НБУ × (1 + `nbu_rate_buffer_pct`)**, буфер **1%** (коригування коливань). Фетч (API vs ручне) уточнимо на збірці.
- `[D3]` ✅ **Курс з історією (effective-dated)** замість одного поточного значення — лікує C2 (старі JP-лоти беруть курс на свою дату). **Увага зміни напряму:** зараз у таблиці `JPY = 3.5` означає «JPY за 1 грн» (ділення). У новій схемі уніфікуємо на **«грн за 1 одиницю валюти»** (множення) — інтуїтивніше і однаково для USD/EUR. Конвертацію старих даних зробимо в імпорті.
- ✅ Валюти: **JPY, USD, EUR** (без CNY) — підтверджено власником.

**Обчислювані (view/функція, не зберігаємо — для аудиту прозоро):**
- `prro_total = goods_cost_uah + forwarding_fee_uah + intl_shipping_uah + local_delivery_uah` (landed) → `prro_unit = /qty`
- `mgmt_total = prro_total × (1 + credit_servicing_pct)` → `mgmt_unit = /qty`

- `[D9]` ✅ Собівартості — **обчислювані в БД-функції/в'юсі**, а не приховані формули аркуша (прозоро, тестовано проти анкерів).
- `[D10]` ✅ **Канонічні статуси партії** (за визначеннями власника, усуває F1/F2):

| code | UA | сенс |
|---|---|---|
| `ordered` | Замовлено | замовлено в постачальника, оплачено; у дорозі до посередника або на його складі (Японія = ZenMarket). Для України — фактично одразу переходить далі. |
| `in_transit` | В дорозі | відправлено зі складу посередника (або від постачальника для України) на основний склад |
| `in_stock` | На складі | фізично на основному складі, можна реалізовувати |
| `selling` | В реалізації | на основному складі, продано ≥1 од з партії |
| `sold` | Продано | повністю розпродано |
| `cancelled` | Скасовано | партію скасовано (✅ підтверджено — лишаємо) |

  - **FIFO/склад** враховує `in_stock + selling + sold` (списання залишків звідси). **Актив-не-склад** = `ordered + in_transit`.
  - **Автостатус** (логіка зі старої CRM): надходить продаж → перша партія по даті споживається; коли зачеплено ≥1 од партії зі `in_stock` → `selling`; коли спожито всю партію → `sold`. Оновлюваний набір = `{in_stock, selling}`.
  - «Партія» = стара назва «лот». «Основний склад» = фізичний склад магазину в Україні, з якого йде відправка покупцям.
  - Старі статуси (`На складі в Японії`, `Виграно`, `Частково продано`) при імпорті → `legacy_status` + мап на нові (`Японія/Виграно → ordered`, `Частково продано → selling`).

| `payment_fee` | numeric(12,2) | комісія оплати (еквайринг/ФОП/післяплата) по рядку |
| `cost_method` | text | `FIFO` / `FIFO+fallback` / `Fallback` / `Відкладено` |
| `cost_audit` | text | трасування партій (як у старій CRM) |
| `cost_fixed_at` | timestamptz NULL | коли зафіксовано COGS |
| `note` | text | |

**Обчислювані (view):**
- `revenue = qty×unit_price − discount_alloc` (= кол. K)
- `prro_cogs = prro_unit×qty`; `mgmt_cogs = mgmt_unit×qty`
- `gross_profit = revenue − prro_cogs` (= кол. U)
- `net_profit = revenue − mgmt_cogs − packaging_alloc − payment_fee − shop_delivery_alloc` (= кол. V)

- `[D11]` ✅ COGS і комісії — **снапшот** на момент, коли продаж стає «реальним» (оплачено/відправлено/отримано), далі незмінні (цілісність обліку). До того — `cost_method='Відкладено'`.
- `[D12]` ✅ Комісії оплати — правило в `payment_types` (які `app_config`-ставки застосувати), рахуються по рядку як зараз (Еквайринг 1.5% / ФОП 0.5%·мін / Післяплата 2%+20).
- `[D14]` ✅ «Реальний продаж» для COGS/звітів = `(Оплачено ∨ Отримано ∨ Відправлено) ∧ не Скасовано/Повернення` (= поточний `isActualSaleForCost_`).

## 2.5 FIFO-COGS (функція)

`fn_fix_sale_cogs(sale_item)`:
1. Партії SKU зі статусом `in_stock/selling/sold`, дата надходження ≤ дати продажу, сорт по даті (партії без дати — найраніші, з логом).
2. «Спожито до» = прийняті продажі цього SKU до цієї дати + списання до цієї дати.
3. Бере потрібну к-сть FIFO, рахує `prro_unit`/`mgmt_unit`; нестачу → fallback (поточна собівартість зі складу/товарів) з **явним флагом**, не тихий 0 (виправляє F4).
4. Знімає снапшот у `sale_item`.

- `[D15]` ✅ Реалізовано як БД-функція + тригер на зміну статусу/складу; нічна звірка-перерахунок (як `updateLotStatuses`).

# STAGE 3 — Mystery box, консумативи, списання

## 3.1 Mystery box — логіка (за визначенням власника)

Бокс збирається з **наявних паків на розсуд власника** (без жорсткого рецепту). Два типи (тільки Pokémon і One Piece):

| тип | вміст | РРЦ | провізорна собівартість |
|---|---|---|---|
| `MBX` | 5 паків, без холо | 700 | **450** |
| `MBX-XL` | 7 паків + 1 холо | 950 | **700** |

Холо-карта = **75 грн** фіксовано (з колекції власника), лише для XL.

**Дві гілки собівартості:**
- **OpenCart-замовлення:** одразу **провізорна стала** собівартість (450/700). Пізніше власник вносить фактичний вміст через форму **списань** → собівартість перераховується на фактичну.
- **Ручні замовлення (вносить власник):** одразу **фактична** собівартість з указаних паків (+ холо для XL + консумативи).
