### 1.2 Інвентаризація костилів
- **K1 — адреса НП текстом.** Language-overrides підмінюють стандартні лейбли: `payment_address.php: entry_address_1 = 'Адреса та номер відділення чи поштомату'`. Користувач руками друкує відділення у звичайний text input. Реальна NP-форма Pinta інжектиться event-ами в `shipping_address`/`register`, але на сторінці не використовується.
- **K2 — поштомат через коментар.** «Для доставки в поштомат: напишіть адресу та номер поштомата в коментарі» — інструкція замість вибору.
- **K3 — Hutko crutch** (`cost => 0` у getQuote) — **рішення owner оновлено: ЗАМІНЮЄТЬСЯ штатним механізмом**. Нова цільова логіка: getQuote повертає реальну вартість з NP API → доставка показується на чекауті і входить у суму замовлення (order_total). Hutko/фіскалізація при цьому отримують суму БЕЗ доставки і зі знижками — це ВЖЕ вбудовано в Hutko extension: `buildRequest` збирає shipping з `order_total` (code='shipping'), при `payment_hutko_shipping_include = 0` (у нас уже так налаштовано) віднімає її від amount, а `normalizeProductsTotal` нормалізує products до цієї суми → чек Checkbox консистентний. Поріг free_from тепер керує реальною вартістю: cart ≥ поріг → shipping cost = 0 («За наш кошт»). ОСОБЛИВА УВАГА в QA: Hutko amount і фіскальний чек на замовленні з доставкою + купоном.
- **K4 — невідома live-зміна.** Текст «Щоб додати нову адресу, перейдіть у розділ "Мої адреси"» є на live, але ВІДСУТНІЙ і в ранковому бекапі, і в r135-патчі. На live є зміни поза контролем. → ST-0.
- **K5 — JS-relabel шар** (r11b): працює постфактум по DOM, ламкий.
- **K6 — вибір «гість/реєстрація»**: stock radio register/guest у `register.twig` є в коді, на скриншоті не видно — ймовірно прихований. Verify у ST-0.

### 1.3 Pinta NP модуль
v1.4.0, інтеграція через OC events (інжекція в HTML-відповідь через str_replace). Локальні патчі: 3 historical + r135 (виконано 2026-06-12, 9 файлів, бекап `_patch_backups/r135_np_module_fix_20260612_073115`). AJAX-довідники: `searchArea/searchCity/searchWarehouse/searchStreet` — локальні таблиці, швидко, без звернень до NP API в рантаймі.

### 1.4 ROOT CAUSE: баг оновлення відділень («Занадто багато запитів»)
`catalog/controller/shipping/reference.php::updateWarehouseReference`:
1. `deleteAll()` — спочатку ВИТИРАЄ всю таблицю відділень;
2. потім тягне NP API `Address/getWarehouses` сторінками по 1000 БЕЗ затримок і БЕЗ retry;
3. в Україні зараз ~35–40 тис. точок (відділення + поштомати) → 35+ запитів підряд → NP rate limit на ~13-й сторінці → цикл рветься;
4. результат: таблиця обрізана на **13 000** записів (13 сторінок × 1000) — saw це і на твоєму скриншоті, і в DB dump;
5. бонус-баг: `reference_warehouse_last_update_datetime` ставиться НАВІТЬ при фейлі — дата виглядає свіжою, дані ні.

Висновок: довідник відділень неповний **з дня встановлення** (збігається з твоїм спостереженням). Бракує ~2/3 точок, насамперед поштоматів. Той самий патерн (deleteAll-first + дата-при-фейлі) — ризик і для інших довідників, але вони менші за обсягом і пролазять під ліміт.

### 1.5 Суміжні механіки (не зламати)
- **Hutko**: amount = order total (cents) з `order_total` після знижок; standalone extension `extension/hutko/`. Checkbox-фіскалізації В КОДІ НЕМАЄ — підв'язана на стороні Hutko-сервісу (verify, питання №2).
- **First15**: `account/register` → `session welcome_coupon_pending = 'First15'`; success.php чистить ключі. Готовий механізм для авто-реєстрації (ST-4).
- **Oferta/agree, різні гілки**: гість — agree-switch у `register.twig` (text_agree); залогінений — agree у `payment_method.twig` + текстовий блок «Натискаючи "Оформити замовлення"...». Зберегти обидві.
- **CRM order-sync**: формат полів адреси замовлення міняти не можна без флага owner.
- Order history: старі замовлення мають address_1 = вільний текст — нові мають лишатись читабельними в тому ж полі.
