## 2. Цільова архітектура (огляд)

```
                    ┌──────────────────────────────┐
   OpenCart (магазин) ──order event──►  Edge Function: order-sync  ──►  Supabase Postgres
   (PHP hook, Codex)    (HTTPS+secret) └──────────────────────────────┘        ▲
                                                                                │
   Власник / браузер ─────►  Next.js (Vercel)  ──►  Repository layer (TS)  ──►  PostgREST / RPC
                              (web-PC, далі mobile)   (єдиний шар доступу)        │
                                                                                 │
   Claude  ───────────────►  Supabase MCP (read-scoped SQL)  ──────────────────►─┘
   Codex   ───────────────►  supabase gen types → точні TS-типи зі схеми
```

**Компоненти і ролі:**

- **Supabase Postgres** — єдине джерело даних CRM. Схема через SQL-міграції (версіонуються в репо).
- **PostgREST (авто-API)** — CRUD поверх таблиць/в'юх; replace Apps Script GET-ендпоінтів.
- **RPC / SQL-функції** — складна логіка (FIFO, summary) як `FUNCTION`/`VIEW`, не в застосунку.
- **Edge Functions (Deno/TS)** — `order-sync` (інтейк OpenCart), пізніше будь-які важкі/планові джоби.
- **Next.js (App Router, TS) на Vercel** — веб-UI, mobile-friendly за дизайном, перший — PC.
- **Repository layer** — `lib/repositories/*`: UI ніколи не звертається до Supabase напряму (див. §6).
- **Supabase Auth** — один акаунт власника (magic-link). RLS: `authenticated` = повний доступ; service-role лише в Edge Functions.

**Середовища:** local Supabase emulator (dev) → Supabase Free (staging/dev-cloud) → Supabase Pro $25/міс (prod при cutover).

## 3. Schema-first: модель даних (чернетка на звірку)

Усі поля виведені з поточних форм/статусів дашборда. **[CONFIRM]** — підтвердити перед першою міграцією.

### 3.1 Довідник товарів

**`products`** (майстер SKU; редагований)
- `sku` text **PK**
- `name` text
- `game` text — MTG / Pokemon / One Piece / Yu-Gi-Oh **[CONFIRM] enum чи довідник**
- `language`, `set_code` text (nullable)
- `gtin` text (nullable) — **тільки якщо власник надасть; не вигадуємо**
- `rrc` numeric (nullable) — РРЦ (рекомендована роздрібна ціна), редагована
- `rrc_updated_at` timestamptz
- `active` boolean, `created_at` timestamptz

> Похідні показники по SKU (sold_30d/60d, margin_pct_60d, max_buy_price, stock-status, coverage) — **не зберігаємо**, рахуємо у в'юхах (§5). У поточній CRM вони теж обчислювані.

### 3.2 Закупки і лоти (FIFO-основа)

**`purchases`** (шапка закупки)
- `id` uuid **PK**
- `supplier_channel` text — `zenmarket_jp` / `supplier_ua` / `other`
- `order_ref` text (nullable для не-ZenMarket), `order_url` text
- `total_cost` numeric — загальна вартість закупки **[CONFIRM] у якій валюті: грн чи валюта каналу?**
- `japan_fees_jpy` numeric (0 для не-JP)
- `ukraine_delivery_uah` numeric
- `status` text — `Замовлено` / `В дорозі` / `На складі UA` / `Скасовано`
- `note` text, `purchased_at` date, `created_at` timestamptz

**`purchase_items`** (= ЛОТ; одиниця FIFO-споживання)
- `id` uuid **PK**, `purchase_id` uuid **FK→purchases**
- `sku` text **FK→products**
- `qty` integer
- `manual_cost` numeric (nullable) — ручне перевизначення собівартості одиниці
- `unit_cost` numeric — **обчислюваний** базис собівартості (алокація total_cost+fees+delivery або manual_cost); формула — предмет §5
- `note` text

> **[CONFIRM] локація лота.** Зараз location визначає `purchases.status` (UA vs «в дорозі»). Історичні статуси «На складі в Японії», «Виграно» (legacy) потрібні для коректного `asset_cost`. Рішення: при імпорті зберігаємо оригінальний legacy-статус у `purchase_items.legacy_status`, а нова робота йде на 4 статуси.

### 3.3 Продажі

**`sales`** (шапка замовлення/продажу)
- `id` uuid **PK**
- `opencart_order_id` text (nullable, **UNIQUE**) — ключ ідемпотентності інтейку
- `channel` text — `OpenCart` / `Telegram` / `OLX` / `Monobazar` / `Вручну` / `Інше`
- `sold_at` date
- `customer_phone`, `customer_name` text
- `payment_type` text — `Контроль оплати ФОП` / `Післяплата фіз` / `За реквізитами` / `Еквайринг`
- `payment_status` text — `Не оплачено` / `Оплачено` / `Повернення` / `Скасовано`
- `order_status` text — `Нове` / `В обробці` / `Відправлено` / `Отримано` / `Скасовано` / `Повернення` / `Передзамовлення`
- `discount` numeric, `packaging_type` text, `custom_packaging_cost` numeric (nullable)
- `shop_delivery` numeric — компенсація доставки
- `post` text (`НП`/`УП`/`Meest`/`Самовивіз`/`Інше`), `ttn` text
- `note` text, `created_at` timestamptz

**`sale_items`**
- `id` uuid **PK**, `sale_id` uuid **FK→sales**
- `sku` text **FK→products**, `qty` integer, `price` numeric (ціна продажу одиниці), `note` text

**`mystery_components`** (розклад mystery-box усередині продажу)
- `id` uuid **PK**, `sale_item_id` uuid **FK→sale_items** **[CONFIRM] прив'язка до line чи до sale**
- `sku` text **FK→products**, `qty` integer, `note` text

### 3.4 Списання

**`writeoffs`**
- `id` uuid **PK**, `type` text (`Власне відкриття`/`Подарунок`/`Промо`/`Брак`/`Тест`/`Втрата`/`Інше`)
- `reason` text, `expected_qty` integer (nullable), `note` text, `written_off_at` date, `created_at` timestamptz

**`writeoff_items`**
- `id` uuid **PK**, `writeoff_id` uuid **FK→writeoffs**, `sku` text **FK→products**, `qty` integer, `note` text

### 3.5 Похідні в'юхи / функції (читання)

| Об'єкт | Призначення |
|---|---|
| `v_stock` | Залишок по SKU = Σ purchase_items.qty (по статусах) − Σ sale_items.qty − Σ writeoff_items.qty; + stock-status (ОК/Мало/Немає) |
| `fn_fifo_cost(sku)` / `v_fifo_cost` | FIFO-споживання лотів, залишкова к-сть і собівартість залишку (UA-only) |
| `v_sku_metrics` | sold_30d, sold_60d, margin_pct_60d, max_buy_price, coverage_days |
| `v_summary` | warehouse_cost, asset_cost, potential_profit_warehouse, asset_potential_profit, MTD revenue/profit |

Логіка цих об'єктів — предмет аудиту §5 (зараз вона «зашита» в `apiSummary_` Apps Script + частково у фронт-JS).

**ER коротко:** `products 1—N purchase_items N—1 purchases` · `products 1—N sale_items N—1 sales` · `sale_items 1—N mystery_components` · `products 1—N writeoff_items N—1 writeoffs`.

## 4. Frontend — обґрунтування вибору

**Рішення: Next.js (App Router, TypeScript) на Vercel.**

Чому, з огляду на заявлені пріоритети (мінімум переробок, без рефакторингів при зростанні, дешево на старті):

- **Рідна інтеграція з Supabase** (офіційний `@supabase/ssr`, auth, типи) — менше клею, менше помилок Codex.
- **Mobile-ready без переписування** — той самий код адаптуємо під телефон responsive-версткою пізніше (відповідає п.4: база закладається зараз, mobile реалізуємо в кінці).
- **Vercel Hobby — безкоштовно**, деплой з git push, прев'ю-середовища. Збігається з ціновою логікою (Free→Pro лише коли треба).
- **Екосистема компонентів** (Tailwind + готові UI-кіти) — швидкий «нормальний дизайн, не таблиця» з вимог UI/UX.
- **Auto-generated TS types зі схеми** — Codex отримує точну структуру (вимога з чернетки §взаємодія).

Розглянута альтернатива — Vite SPA (легше, без SSR). Відхилено: гірша інтеграція auth/SSR із Supabase і більше ручного клею; виграш у «легкості» не вартий додаткової роботи на масштабі. **[CONFIRM]** якщо власник хоче мінімалізм — залишаю Vite SPA як план Б.

## 5. Аудит фінансових формул (повний перелік на перевірку)

Власник просив переглянути **всі** розрахунки від собівартості до кінцевих KPI — раптом «перемудрили або не врахували». Нижче — повний інвентар того, що зараз рахується (з джерел), і конкретні питання на коректність. Це окремий робочий блок Фази 0/1: Claude піднімає поточні формули, складає звідну фінмодель, власник підтверджує, далі Codex реалізує в SQL.

**Анкери для звірки (значення станом на 2026-06-19, `DASH-001`):** `warehouse_cost=26386.27`, non-UA asset=44647.52, `asset_cost=71033.79`, `potential_profit_warehouse=25304.42`, `asset_potential_profit=60724.53`. Зараз будуть інші, але метод звірки — відтворити ці числа з тих самих даних.

| # | Розрахунок | Поточна логіка (з джерел) | Питання на коректність |
|---|---|---|---|
| F1 | Собівартість одиниці лота | `manual_cost` override АБО алокація `total_cost` | Як саме алокується total_cost між SKU — за к-стю чи за вартістю? Чи входять `japan_fees_jpy` і `ukraine_delivery_uah` у собівартість? |
| F2 | Валюта/курс | `total_cost`, `japan_fees_jpy` (JPY), `ukraine_delivery_uah` (UAH) | JPY→UAH: за яким курсом і на яку дату (закупка/оплата/фікс)? Де зберігається курс? Зараз курс ніде не видно — **ризик некоректної собівартості JP-лотів** |
| F3 | Коефіцієнт ×1.05 | «коригована собівартість = cost×1.05» у profit і в margin% | Що це за 5% (комісія/податок/опер.витрати)? Актуальний? Має бути конфіг, не константа в коді? |
| F4 | FIFO-споживання | FIFO по лотах; transit/JP/won виключені зі складу; залишки в old-rows `Продано` не дропаються; враховує негативні позиції | Порядок лотів (дата?), коректність часткового продажу, чи правильні виключення статусів |
| F5 | warehouse_cost | FIFO-собівартість UA-залишку | Тільки UA-склад? Збігається з визначенням власника? |
| F6 | asset_cost | UA + в дорозі + JP + виграно | Усі нереалізовані; legacy-статуси враховані? |
| F7 | potential_profit | `(RRC − cost×1.05) × remaining_qty`; лоти без RRC не дають прибутку | Чи коректно ігнорувати лоти без РРЦ? Як заповнювати прогалини РРЦ? |
| F8 | margin% | `profit / (cost×1.05 + profit) × 100` | Це маржа від виручки; визначення збігається з очікуваним? Узгоджено з F3 |
| F9 | Фактичні revenue/profit (MTD) | orders, revenue, profit за місяць | Чи нетяться `discount`, `custom_packaging_cost`, `shop_delivery`? Як враховуються `Повернення`/`Скасовано`? |
| F10 | Estimated margin | при відсутності 60д продажів: `max_buy×1.25` як ціна продажу | Евристика 1.25 актуальна? Не спотворює агрегати? |
| F11 | Mystery-box | склад компонентів у `mystery_components` | Як розноситься собівартість компонентів і виручка mystery-box? |
| F12 | Coverage / швидкість | `stock / (sold30/30 або sold60/60)` | Вікна 30/60д ок? Поведінка за нульових продажів |

**Вихід блоку:** `plans/crm-financial-model_<date>.md` — звідна, перевірена фінмодель з однозначними формулами → основа для `v_summary`/`fn_fifo_cost`. Без цього Codex не пише розрахункові в'юхи.

## 6. Repository Pattern (специфікація)

Мета з чернетки: змінюється один шар при зміні структури/платформи, не весь проєкт.

```
lib/
  supabase/client.ts        // єдина точка ініціалізації клієнта (server/browser)
  types/database.ts         // supabase gen types (авто, не редагувати руками)
  repositories/
    sales.repo.ts           // listSales(), getSale(), addSale(payload), updateSaleStatus()
    purchases.repo.ts       // lots, addPurchase(), batchUpdateStatus()
    writeoffs.repo.ts
    products.repo.ts        // RRC edit, sku list
    analytics.repo.ts       // getSummary(), getStock(), getSkuMetrics()  → читають в'юхи/RPC
  domain/                   // чисті типи домену (Sale, Lot, Summary) — незалежні від Supabase
```

Правила: UI/компоненти імпортують **тільки** `repositories/*` і `domain/*`; прямих `supabase.from(...)` поза repo немає. Кожен публічний метод приймає/повертає `domain`-типи, всередині мапить на таблиці. Це дає точку заміни (інша БД/інший бекенд) і чисту поверхню для Codex-хендофів.

## 7. Order pipeline: OpenCart → Supabase (заміна Apps Script)

Зараз: OpenCart шле замовлення в Apps Script `doPost` → `upsertOpenCartOrder_`. Ціль: прямий інтейк.

**Дизайн:**
1. **OpenCart hook (PHP, Codex)** — на подію `order add` / зміну статусу формує JSON і робить `POST` на Edge Function з shared-secret заголовком. **Fire-and-forget**: помилка інтейку **ніколи не блокує checkout** (try/catch, короткий таймаут, лог).
2. **Edge Function `order-sync` (TS)** — перевіряє secret, валідує payload, `upsert` у `sales`+`sale_items` за `opencart_order_id` (ідемпотентно). Повертає 200 швидко.
3. **Ретраї** — за потреби легка чергова таблиця `sync_failures` для ручного/планового переграшу.

**Ризик — високий (зачіпає живі замовлення):** обов'язковий smoke checkout (§9). Поки pipeline не звірений — Apps Script лишається паралельно як резерв, вимикаємо після підтвердження.

**[CONFIRM]** які саме події OpenCart ловимо (тільки нове замовлення, чи й зміни статусу/оплати), і чи є доступ правити код магазину для hook.

## 8. План реалізації (фази, розподіл, зусилля)

Позначки: **C**=Claude (план/хендоф/ревʼю/SQL-специфікація) · **X**=Codex (код) · **M**=Manual (власник: Supabase-проєкт, секрети, деплой, прогон).

### Фаза 0 — Фундамент (schema-first + скелет)
- **C:** фіналізувати схему §3 і фінмодель §5; репо-патерн §6; хендофи.
- **X:** SQL-міграції (таблиці §3), Next.js-скелет, repo-шар, `gen types`, local Supabase emulator.
- **M:** створити Supabase-проєкт (Free), поставити Supabase CLI, тримати секрети, перший deploy на Vercel.
- **Вихід:** порожня, але повна БД + застосунок, що читає Supabase. **DoD:** міграції застосовуються чисто, типи генеруються, emulator піднімається.

### Фаза 1 — Ядро CRM (web-PC), паралельно зі Sheets
- **C:** специфікації в'юх (`v_summary`, `fn_fifo_cost`, `v_stock`, `v_sku_metrics`) за §5; UX-екрани.
- **X:** read-екрани (summary-картки, замовлення, склад, SKU, клієнти); write-форми (продаж/закупка/списання/редагування РРЦ) через repo; розрахункові в'юхи/RPC.
- **X+M:** разовий імпорт історії зі Sheets (CSV→import-скрипт), **звірка KPI проти поточного дашборда** (анкери §5).
- **Вихід:** робоча CRM у браузері з реальними даними. **DoD:** summary-числа збігаються з поточними в межах копійок; усі CRUD працюють.

### Фаза 2 — Order pipeline + автоматизації
- **C:** хендоф `order-sync` + OpenCart hook; checkout smoke-план (skill `bs-checkout-smoke`).
- **X:** Edge Function `order-sync`, PHP-hook у OpenCart, idемпотентний upsert; stock-alerts як планова функція за потреби.
- **M:** деплой hook на магазин, прогін smoke, підтвердження → **ретайр Apps Script**.
- **Вихід:** живі замовлення ллються прямо в Supabase. **DoD:** тестове замовлення зʼявляється в CRM без впливу на checkout.

### Фаза 3 — Mobile + поліш (в останню чергу)
- **C+X:** responsive/mobile-вʼюхи поверх готового web (база закладена з Фази 0).
- **Опційно:** AI-функції (Gemini) / BigQuery — відкладено до реальної потреби.
- **M:** власник вирішує scope мобільного після готового web + дизайну.

**Оцінка зусиль** дам у годинах/спринтах **після підтвердження §3 і §5** — обсяг прямо залежить від кількості розрахункових вʼюх і складності FIFO. До підтвердження будь-яка оцінка була б вгадуванням.

## 9. Ризики та smoke-тести

| Ризик | Рівень | Мітигація / тест |

## 10. Взаємодія Claude + Codex з новою CRM

- **Claude:** Supabase MCP з read-scoped доступом — точні SQL замість читання sheet-діапазонів; менше токенів, точніші відповіді. Запис у CRM Claude **не** робить автоматично (як і зараз: правки — через підтверджений потік).
- **Codex:** `supabase gen types typescript` → `lib/types/database.ts`. Codex має точну структуру, не вгадує. Хендофи Claude посилаються на конкретні таблиці/repo-методи.
- **Секрети:** service-role/secret order-sync — лише в Supabase/Edge env і локальному secure-контексті; ніколи в чатах, документах, логах, скриншотах.

## 12. Уточнення після читання коду Apps Script (2026-06-26)

Прочитано `Booster Shop CRM — облік товарів - Apps_Script_код.csv` (батьківська папка). Модель багатша за чернетку — фіксую факти і вирішені питання.

### 12.1 Вирішені [CONFIRM]
- `purchases.total_cost` — у **грн (UAH)**. Лінійна собівартість алокується пропорційно кількості або задається `manual_cost`. (знято)
- Mystery-box — у коді жорстко **рівно 5 компонентів на 1 бокс** (`componentQty == mysteryQty*5`). (лишаю; підтвердити лише, чи правило стале)
- Локація лота: `Замовлено / В дорозі / На складі в Японії / Виграно` = **актив, але не склад UA**; `На складі UA / На складі / Частково продано / Продано` = склад (FIFO). При імпорті зберігаємо оригінальний статус, інакше `asset_cost` поламається. (рішення: зберігаємо як є)

### 12.2 Нові сутності (не було в §3) — додати в схему
- **`currency_rates`** (`Курси`): курс JPY за 1 грн, одне поточне значення; fallback 3.5. **Ризик:** курс застосовується на момент вводу, історичного немає → собівартість старих JP-лотів залежить від курсу на дату вводу (аудит F2).
- **`expenses`** (`Витрати`): прямі витрати, опційно прив'язані до замовлення (`linked=так`, `order_ref`). Входять у собівартість mystery-box.
- **`consumables`**: юніт-вартість видаткових (`Стікер лого+QR`, `Блайнд-пакет для картки`, `Наліпка Mystery Box`) + типи паковання з цінами (`getPackagingCost_`). Входять у собівартість продажу.

### 12.3 Дві собівартості (важливо)
Лот/продаж мають **дві бази**: **ПРРО** (landed = товар+JP+UA доставка) і **управлінська** (= ПРРО × 1.06, де 6% — кредитне обслуговування з `Налаштування`). Прибуток = `(РРЦ − управлінська×1.05)×qty`, де ×1.05 — свідомий дисконт очікуваного доходу (деталі — `crm-financial-model_2026-06-26.md §K`). Нова схема несе обидві бази + **обидва коефіцієнти як конфіги** (`credit_servicing`, `expected_income_haircut`): `purchase_items.prro_unit_cost`/`mgmt_unit_cost`; `sale_items.cogs_prro`/`cogs_mgmt`.

### 12.4 Майстер-дашборд НЕ зайвий (корекція чернетки)
`apiSummary_` читає **місячну виручку/прибуток із `Звіт_Продажів`** і дані якості з `Якість_Даних` (файл «Автоматизація»/AUTO). Його логіка (агрегати продажів, MTD revenue/profit) **переноситься** в Supabase (views/функції), а не відкидається.

### 12.5 Order pipeline — які події ловити (відповідь на п.3)
Рекомендований мінімум: (1) **нове замовлення** → insert якщо `opencart_order_id` ще немає; (2) **зміна статусу оплати**; (3) **зміна статусу замовлення**; (4) **ТТН** додано/змінено.
Правила: ідемпотентність за `opencart_order_id`; **не перезаписувати поля, які власник правив у CRM вручну**; не чіпати вже завершені в CRM (Отримано+Оплачено); **виключити тестові замовлення власника** за whitelist — **[потрібно: за чим відрізняти тестові]**.

### 12.6 Наступний deliverable
`plans/crm-financial-model_<date>.md` — кожна формула від собівартості до KPI (F1–F12 + знахідки вище) з вердиктом «коректно / під питанням / помилка». Це і є перевірка обліку, яку ти просив.

## 13. Уточнення вимог round-2 (2026-06-26)

### 13.1 Мультивалютність по кожному полю
Зараз комісія ZenMarket + доставка по Японії + доставка в Україну з Японії вводяться в JPY і авто-конвертуються в грн (один поточний курс). Нова вимога: **для кожного грошового поля — вибір валюти (грн / JPY / USD / EUR) з дефолтом під сценарій і можливістю змінити в дропдауні.**
- Таблиця `currency_rates`: курси JPY/USD/EUR → грн, з датою.
- **Зберігати по кожному полю трійку:** `amount_original` + `currency` + `rate_used` + `amount_uah` (конвертоване). Це лікує поточну ваду «історичного курсу немає».
- Дефолти (приклад ZenMarket): вартість лоту — **грн**; доставка в Україну з Японії і доставка+комісія ZenMarket — **JPY**. Дефолти конфігуруються per supplier_channel.

### 13.2 Редизайн Mystery Box (нова логіка з нуля)
Поточне (legacy): 1 SKU mystery box = 5 паків + 1 карта з колекції власника, РРЦ 750 (PKM-JP-MIX-MBX, OP-JP-MIX-MBX).
**Нова логіка (тільки Pokémon і One Piece):**
- **MBX** — Mystery Box = **5 паків, без +1 холо**, РРЦ **700 грн**.
- **MBX-XL** — Mystery Box XL = **7 паків + 1 холо карта**, РРЦ **950 грн**.

Модель: `bundle_recipes` — bundle-SKU → склад (pack-SKU × к-сть + опційно холо-карта). Холо-карта береться з власної колекції, **база собівартості холо = 75 грн** (фіксована). Нова CRM рахує mystery-box за рецептом, не за жорстким «×5». Legacy-формат (5+1, РРЦ 750) — лише для імпорту історії.

### 13.3 Категорії / асортимент — без жорсткого enum
Інших ККГ поки немає, але **аксесуари вже додані** (ACC-001…006, бренди Generic / Games 7 Days) і лінійка розширюється + **англомовна продукція** по наявних ККГ.
- `category` — довідник (Booster / Booster Box / Blister / Collection Set / Mystery Box / Accessory / …), **не enum**.
- `brand` і `language` (UA / EN / JP / KR …) — окремі поля. Схема НЕ хардкодить 4 гри і не припускає лише JP.
- РРЦ-аркуш має дві ціни: ручна `РРЦ, грн` і `Динамічна РРЦ` — у новій схемі розвести явно (яка канонічна для прибутку — рішення в аудиті §F).

### 13.4 Order pipeline — фінально
- Pipeline **тягне ТІЛЬКИ нові замовлення** (insert). Зміна статусу оплати/замовлення, ТТН — власник робить **вручну**, як зараз (pipeline їх не чіпає).
- **Тест-фільтр (будь-який тригер окремо → НЕ тягнути замовлення):** прізвище `Леусенко`; телефон `0991119279`; слова `Тест` або `TEST` у назві SKU **або** в описі SKU **або** в примітці до замовлення.
