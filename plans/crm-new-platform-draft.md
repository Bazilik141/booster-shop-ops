## Мета

Замінити розрізнені інструменти керування бізнесом єдиною онлайн-платформою з зручним UI, доступом з мобільного та прямою інтеграцією з Claude і Codex.

---

## Поточний стан (від чого відштовхуємось)

| Інструмент | Роль | Проблема |
|---|---|---|
| Google Sheets CRM | База замовлень, клієнтів, статусів | Повільно, квоти Apps Script, обмежені запити |
| Google Sheets майстер-дашборд | Обрахунки, агрегати | Ймовірно вже не потрібен — логіка переїде на бекенд |
| Apps Script API | Ендпоінти для Claude (`action=summary`, `action=orders` тощо) | 6-хв ліміт виконання, важко розширювати |
| Notion roadmap | Управління задачами та роадмапою | Залишається як є, не мігрує |
| Локальний дашборд-сервер (`booster-dashboard.html`) | UI для CRM та roadmap-дзеркало | Доступний лише з домашнього ПК, не мобільний |

**Ключове обмеження зараз:** швидкість роботи відчутно лімітована Apps Script. Все що виходить за рамки простих CRUD-операцій — болісно повільно або неможливо.

## Прийняті рішення

### Платформа: Supabase (PostgreSQL)

**Відхилено:** Firebase/Google stack  
**Причина відхилення:** після уточнення пріоритетів виявилось, що Firebase програє по ключових критеріях:
- NoSQL потребує переписування при зміні схеми
- Pay-as-you-go непередбачувані витрати
- Vendor lock-in важко усунути

**Чому Supabase:**
- SQL (PostgreSQL) — зміна схеми через `ALTER TABLE`, без ручних міграцій документів
- Фіксована ціна: Free → $25/міс Pro (без сюрпризів)
- Open source, мінімальний vendor lock-in
- Supabase MCP — Claude підключається напряму до бази
- Auto-generated TypeScript types — Codex отримує точну структуру, менше помилок

### Пріоритети (в порядку важливості)

1. Мінімум зусиль — не переписувати все знову і знову
2. Не мати глобальних рефакторингів при зростанні
3. Менше фінансових ресурсів на старті

### Ціновий план

- **Розробка і тестування:** Free tier (500MB БД, 5GB трафік — вистачить роками на поточних обсягах)
- **Продакшн (щоденне використання):** $25/міс Pro — одразу як CRM стає робочим інструментом
- **Головна причина переходу на Pro:** Free tier паузить проект після 7 днів неактивності — неприйнятно для CRM

### Архітектурні рішення

**Repository Pattern (обов'язково)**  
Вся логіка роботи з базою ізолюється в окремому шарі коду. При потребі зміни структури чи навіть платформи — міняється один шар, не весь проект.

**Schema-first підхід (обов'язково)**  
Перший крок — проектуємо всі таблиці і зв'язки до першого рядка коду. Власник описує сценарії, Claude будує схему, підтверджуємо — і тільки тоді код.

**Dev emulator**  
Локальне середовище Supabase для тестів. 30 хвилин налаштування один раз, далі — безпечні експерименти без торкання продакшн-бази.

**Поетапний запуск**  
- Фаза 1: Supabase DB + веб-фронтенд (замінює Sheets + локальний дашборд)
- Фаза 2: автоматизації, Edge Functions (замінює Apps Script)
- Фаза 3 (опційно): Gemini API для AI-функцій якщо виникне потреба

**BigQuery — відкладено**  
Потрібен тільки для складної аналітики. Поки не потрібен.

## Вимоги до UI/UX

- Не таблиця — кастомний дашборд з нормальним дизайном
- Доступ онлайн з будь-якого пристрою
- Частина функцій доступна з мобільного (перегляд замовлень, статуси)
- Швидкий відгук (не як Sheets)
- Можливість розширювати нові екрани і функції без переписування бази

---

## Взаємодія Claude + Codex з новою CRM

**Claude (через Supabase MCP):**  
Прямий доступ до бази без Apps Script прошарку. Точні SQL-запити → JSON відповідь. Значно менше токенів ніж читання цілих sheet-діапазонів.

**Codex:**  
Supabase генерує TypeScript types зі схеми автоматично. Codex має точну структуру даних і не вгадує — менше помилок, менше ітерацій на патч.
