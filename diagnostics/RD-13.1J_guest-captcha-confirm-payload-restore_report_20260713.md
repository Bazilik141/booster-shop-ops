# Codex Report — RD-13.1J: restore guest CAPTCHA confirm payload

Date: 2026-07-13

## Scope

Implemented the RD-13.1C client contract only in `checkout.twig`: the explicit place-order action now sends the guest form and `g-recaptcha-response` to `checkout/confirm.confirm` via POST. A 422 response bearing `data-bs-captcha-error` resets the widget and shows the CAPTCHA-specific error without a page reload.

`catalog/controller/checkout/confirm.php` is deliberately untouched. The supplied live copy already contains both required server safeguards: RD-13.1C returns 422 before confirmation for a failed guest CAPTCHA, and ST-2b6e allows order writing only through `confirm()` / `index(true)`.

## Phase 0 evidence and root cause

- The supplied live `checkout.twig` had one actual `confirm.confirm` caller. It was the legacy jQuery `.load(...)` GET at the deferred place-order handler, with no request payload.
- The supplied live `confirm.php` contains the RD-13.1C guest validation and returns `<div ... data-bs-captcha-error="1">` with HTTP 422. This exactly explains the observed live 422 and generic fallback message.
- The live backup manifest confirms RD-13.1C through RD-13.1I and ST-2b6e backups are present. The currently live Twig was last written at the RD-13.1I timestamp, but the supplied archive contains only the backup manifest, not the historical Twig copies. Therefore the specific operation that removed the RD-13.1C client half is **undetermined**; it is not asserted as a fact here.
- The RD-13.1D–I patch sources do not replace or introduce a `confirm.confirm` loader. They preserve their input Twig while making visual/cache-buster edits, so they are not direct evidence of the overwrite action.

Root cause of the active failure is established: the RD-13.1C server and client halves were out of parity. The server required a CAPTCHA token while the live client made an empty GET request.
