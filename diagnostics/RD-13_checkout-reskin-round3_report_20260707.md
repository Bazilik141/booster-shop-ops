# Claude Report — RD-13: checkout reskin round 3

Date: 2026-07-07
Author: Claude (не Codex — патч зібрано напряму після діагностики live)

## 1. Short conclusion

Round 1–2 виглядали "провальними" не лише через якість виконання. Дві реальні
причини, знайдені на живому сайті:

1. **Stale CSS cache.** Round 1 і round 2 задеплоєні під одним
   cache-buster `?v=rd13-20260706`. Браузери (і кеш хостингу) досі віддають
   round-1 CSS — жодна правка round 2 не була видима. Перевірено в
   devtools: завантажений stylesheet не містить round-2 правил, тоді як на
   сервері вони є.
2. **Регресія P0-1 у round 2.** Правило ховає `.bs-confirm-deferred-summary`
   цілком, а всередині цього вузла — вся таблиця товарів і підсумків. Із
   застосованим round-2 CSS картка "Замовлення" втрачає товари, суму та
   прогрес доставки. Помилка була ще у формулюванні review-handoff P0-1 —
   Codex виконав його дослівно.

Round 3 виправляє обидві причини + доводить подачу до мокапа без дотику до
логіки: кастомний рендер підсумку замовлення, чисті назви способів доставки
та адрес, стилі для блоку адреси залогіненого користувача.

## 5. What the patch does

Files: `checkout.twig`, `boostershop-ds.css`, `checkout-reskin.js` (3 файли,
бекапи в `_patch_backups/`, sha256-гейти, dry-run режим `--dry-run`).

1. **checkout.twig** — тільки bump версій: `?v=rd13-20260706` →
   `?v=rd13r3-20260707` (CSS+JS). Це змушує всі браузери й кеш хостингу
   забрати свіжі файли. Більше в twig нічого не чіпається.
2. **boostershop-ds.css** — round-2 правило P0-1 звужено: ховається лише
   `.bs-confirm-method-summary` (дубль-рекап "Доставка/Оплата"); таблиця
   товарів більше не ховається. Доданий round-3 блок: стилі item-рядків,
   підсумків, порядок блоків у "Замовленні" через flex-order, чистка
   легенд/статусів, стилі адресного блоку, chevron тільки на мобільному.
3. **checkout-reskin.js** — повна заміна файлу (round 3):
   - `buildSummaryView()` — будує підсумок з stock-таблиці confirm:
     товари (скрол після 3 позицій) → блок "Доставка" з прогрес-баром
     безкоштовної доставки (RD13-STUB, поріг 2000, рахується від "До
     сплати", тобто після знижки — за V2 спекою) → Сума товарів / Знижка /
     До сплати. Таблиця лишається в DOM прихованою як джерело даних —
     якщо JS не спрацює, показується stock-таблиця (graceful degradation).
   - Порядок у картці: товари → доставка+прогрес → суми → промокод (stub) →
     коментар → згода/opt-in (гість) → hint + CTA. **Кнопку CTA не
     переміщено** — її делегований обробник вимагає, щоб вона лишалась
     всередині `#checkout-confirm`; порядок вирішено flex-order.
   - `relabelShippingMethods()` — «Нова пошта: доставка у поштомат - За
     тарифами Нової Пошти (≈ ₴74)» → «Нова пошта — поштомат» + підрядок.
     Тільки видимий label; value/data-label/hidden inputs не змінюються.
   - `restyleSavedAddress()` — "Додати нову адресу Нової пошти" → "+ Інша
     адреса", "Я хочу використовувати існуючу адресу" → "Збережена адреса";
     в options збереженої адреси прибирається префікс з ПІБ. Display-only.
   - Видалені `relabelTotals()` і `renderFreeShippingStub()` (замінені
     `buildSummaryView()`), решта round-2 функцій збережені як були.

## 6. Deploy (manual, owner)

Контракт r3a (за ревю Codex): патч ідемпотентний (повторний запуск →
`already_applied=yes`, exit 0), сам чистить кеш OpenCart через
`config.php`/`DIR_CACHE` (data + template cache, без хардкод-шляхів) і сам
видаляється після `done=ok` (`--keep-self`, щоб лишити).

1. Залити `RD-13_checkout-reskin-round3_20260707.php` в корінь сайту (поруч
   з index.php).
2. `php RD-13_checkout-reskin-round3_20260707.php . --dry-run` — усі
   precheck/postcheck `ok`, `done=dry-run`.
3. `php RD-13_checkout-reskin-round3_20260707.php .` — `done=ok`,
   `cache_clear=ok:...`, `self_delete=ok`.
4. LiteSpeed/CDN кеш хостингу, якщо активний, почистити з cPanel (патч
   чистить тільки кеш OpenCart).
5. Якщо `sha256_mismatch` — файл на сервері відрізняється від копій, з яких
   зібраний патч: перекачати свіжі копії, повідомити — перезберу гейти.
6. Якщо `partially_applied:...` — відновити всі 3 файли з `_patch_backups/`
   і запустити знову.
7. Rollback: відновити 3 файли з надрукованої директорії `_patch_backups/`.

## Known limitations (не баги round 3)

- Мокап показує 3 способи доставки; live для збереженої адреси віддає один
  (quote модуля НП). Це дані бекенда, не подача — out of scope.
- Мокап показує мініатюри товарів у підсумку; у stock confirm-таблиці
  зображень немає — без контролера їх не додати. Відкладено.
- Картка "Отримувач" для залогіненого не рендериться (stock не віддає
  register/payment_address) — как у round 1–2, потребує контролера.
- 4-й метод "Оплата при доставці" (дубль) — конфіг платіжних модулів, P2.
- Breadcrumb "Оформлення замовлення" vs H1 "Оформити замовлення" — P2.
