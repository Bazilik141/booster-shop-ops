# RD-13.1C — final CAPTCHA gate for guest checkout

## Scope

Guest checkout only. A solved reCAPTCHA response is now passed to the trusted
`confirm.confirm` request and revalidated by the server before the confirmation
markup (and therefore the payment button/order creation path) is returned.

## Files touched

- `catalog/view/template/checkout/checkout.twig`
- `catalog/controller/checkout/confirm.php`

No database, customer profile, address-book, Nova Poshta, or payment-module
data changes are made.

## Validation

- dry run: `dry_run=ok`
- patch syntax: `php -l` passed
- changed PHP: `confirm.php` passed `php -l`
- staged replay: `done=ok`, backups created, self-delete verified
- repeat replay: `already_applied=yes`

## Rollback

The runner writes original copies before changes to:
`_patch_backups/RD-13.1C_guest_captcha_final_gate_20260712_<timestamp>/`.
Restore both target files from that directory if a rollback is needed.

## Run

```bash
cd ~/public_html || exit
php RD-13.1C_guest_captcha_final_gate_20260712.php && php -r 'require "config.php"; foreach (glob(DIR_CACHE . "cache.*") ?: [] as $f) if (is_file($f)) @unlink($f); foreach (glob(DIR_CACHE . "template/*") ?: [] as $f) if (is_file($f)) @unlink($f); echo "cache cleared\n";'
```

## Post-deploy QA

1. Guest, CAPTCHA not solved: click “Підтвердження замовлення” — show CAPTCHA
   prompt and no order is created.
2. Guest, CAPTCHA solved: one click proceeds to the real payment confirmation.
3. Logged-in customer: checkout remains unchanged.
4. Confirm that no duplicate request/order appears after one valid click.
