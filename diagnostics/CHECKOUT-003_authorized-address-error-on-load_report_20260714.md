# Codex Report — CHECKOUT-003: suppress invalid NP-address autosave on initial load

Date: 2026-07-14

## Scope

The supplied live archive proves that the authorised checkout error is caused before the customer touches the form. This patch changes only the client-side autosave timing; it does not relax validation and does not alter order creation, payment, Nova Poshta API calls, or the database.

Root cause, from the current files:

1. `checkout.twig` calls `updateRecipientMode()` and, after `hydrateNpRefsFromHidden()`, schedules `checkout/register.save` while the checkout is initialising.
2. `bsCheckoutNpIsComplete()` accepts the hidden NP refs. The refs may still be non-empty while the visible city/warehouse fields are blank.
3. `bsCheckoutSyncNpFields()` serialises the visible values, so the request contains an empty `shipping_city` / `shipping_address_1`.
4. `catalog/controller/checkout/register.php` correctly rejects those fields; `surfaceRegisterErrors()` renders the exact red banner from the screenshot.
5. The NP module also programmatically triggers the delivery-type `change` event during its initial render, which reaches the generic register autosave listener.

The fix removes all three initial-load scheduling paths, while retaining autosave after actual recipient or NP-field interaction.

- No DB or server-validation change. The existing server validation remains the final guard.
- The only intended behavioural change is that restored/stale NP values never cause a background write until the customer interacts with the recipient or NP controls.
- Because the patch touches the module's checkout JS template, clear the template/cache entries and perform the two authorised NP interactions above before broad checkout QA.
