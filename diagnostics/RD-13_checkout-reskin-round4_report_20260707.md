# Claude Report — RD-13: checkout reskin round 4

Date: 2026-07-07

## 1. Short conclusion

Round 4 закриває всі 12 зауважень власника з ревю round 3 (гість/авторизований,
мобайл/десктоп). Усе — presentation-only: ендпоінти, значення полів, платіжна
логіка, trusted-click gate і double-submit guard не змінені.

## 2. Що зроблено (мапа на зауваження)

Гість:
1. Хвіст (коментар → оферта-чекбокс → "Зберегти дані"-toggle → CTA) — один
   компактний блок; оферта тепер квадратний чекбокс, не toggle.
2. Капча ("Перевірка безпеки") + підписка на новини — під кнопкою CTA.
3. Фото товарів у підсумку — з мінікошика хедера (та сама сесія, без нових
   запитів; якщо збігу немає — рядок без фото).
4. Мобайл-порядок: Замовлення (згорнуте; товари+прогрес+суми+промокод
   всередині) → Отримувач → Доставка → Оплата → хвіст окремою карткою внизу.
   #checkout-confirm переміщається ЦІЛИМ (делегований обробник CTA живе на
   `#checkout-confirm [data-bs-deferred-confirm]` — кнопка з нього не виймається).
5. З "Отримувача" прибрано: слот капчі, легенду "Доставка Новою поштою",
   блок "Отримувач — інша особа" (сховано CSS, дефолтна логіка незмінна),
   стокову кнопку реєстрації.
6. Сітка Отримувача: Ім'я|Прізвище, По батькові|Телефон, E-mail на всю ширину.
7. "Тип доставки"-dropdown прибрано з очей; натомість 3 radio-пункти першими
   в Доставці (у відділення / кур'єром / поштомат) — презентаційний проксі
   над `#input-shipping-novaposhta-type` (change іде через штатний
   делегований обробник модуля Pinta NP). Список quote-методів прихований
   (зараз один метод, автовибір працює як раніше).
8. Порожня Оплата: inline-статус знову показується, коли список методів
   порожній (r3 ховав його глобально — гість бачив пусту картку).

Авторизований:
9. Recap-картка "Отримувач" (ім'я з даних, що вже є на сторінці). Телефон
   недоступний у DOM — див. Known limitations.
10. Доставка: 3 radio типу (тип підтягується зі збереженої адреси) +
    очищений select адрес (без ПІБ-префікса і ", Ukraine") + лінки
    "+ Інша адреса" / "← Використати збережену адресу" (проксі над
    існуючими радіо shipping_existing).
11. Мобайл: промокод всередині згорнутого "Замовлення", хвіст внизу.
12. Summary Доставки: "Нова пошта · поштомат №49489" / "· відділення №22" /
    "· адреса вул. …" — парситься зі збереженої адреси або введених полів.

Checkout = рух грошей. Обов'язково повний цикл після деплою:
- [ ] Гість: COD → замовлення створене (капча з нової позиції блокує сабміт
      до проходження).
- [ ] Гість + "Зберегти дані" ON → акаунт + magic-link + замовлення.
- [ ] Авторизований: Hutko (мін. сума) → success; Checkbox чек як раніше.
- [ ] IBAN → замовлення + реквізити.
- [ ] Radio типу доставки: обрати кур'єра → ввести адресу → замовлення
      використовує НОВУ адресу, не збережену.
- [ ] Подвійний клік CTA → одне замовлення.

## Known limitations

- Телефон/e-mail авторизованого немає в DOM сторінки — recap показує лише
  ім'я. Повний recap = мінізміна контролера (передати customer telephone у
  data) — окремий low-risk тікет, якщо потрібно.
- Список quote-методів прихований: якщо колись з'явиться другий перевізник,
  його вибір треба буде повернути (зараз метод один і автообирається).
- Фото в підсумку залежать від збігу назви товару з мінікошиком; без збігу
  рядок рендериться без фото.
