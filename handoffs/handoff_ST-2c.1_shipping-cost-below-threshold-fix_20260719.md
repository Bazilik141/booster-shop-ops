# ST-2c.1 — Fix: real NP shipping cost still 0 below the free-shipping threshold

## 1. Task ID
ST-2c.1_shipping-cost-below-threshold-fix (follow-up на ST-2c_real-np-shipping-cost_20260719, вже задеплоєний на живий сервер)

## 2. Context
ST-2c Part A задеплоєний. Власник протестував вручну на живому сайті:
- Кошик ВИЩЕ порогу 2000 грн → коректно показує "Безкоштовно" (ця гілка логіки працює).
- Кошик НИЖЧЕ порогу 2000 грн → доставка досі показує ₴0.00 — так само, як стара заглушка, яку цей патч мав прибрати.

Причина видна прямо в коді вже задеплоєного патчу (`patches/ST-2c_real-np-shipping-cost_20260719.php`), не здогадка.

Патч замінив обидва місця `'cost' => 0,` на `'cost' => $this->getBoosterShippingCost($shipping_cost_default),` і додав метод:

```php
private function getBoosterShippingCost($shipping_cost_default) {
    if ($this->getBoosterCartTotalUah() >= $this->getBoosterFreeShippingFromUah()) {
        return 0.0;
    }
    return is_numeric($shipping_cost_default) && (float)$shipping_cost_default > 0 ? (float)$shipping_cost_default : 0.0;
}
```

Гілка "вище порогу" самодостатня (просто звіряє суму кошика з порогом) — тому вона працює. Гілка "нижче порогу" повністю залежить від `$shipping_cost_default` — а цю змінну патч ніде не оголошує і не перевіряє. Він просто припускає, що вона вже існує в обох місцях виклику (склад і адресна доставка) і вже містить реальний тариф з `getDocumentPrice()` (діагностика ST-2c Phase 0 підтвердила, що `getDocumentPrice()` вже викликається в обох місцях, але його результат раніше відкидався і завжди повертався 0).

Якщо в реальному файлі змінна з реальним тарифом називається інакше (або не в тій області видимості на момент виклику) — `is_numeric()` на неіснуючій змінній поверне false, і функція мовчки впаде на `: 0.0`. Жодної фатальної помилки, жодного попередження, що зупинило б деплой — `php -l` перевіряє тільки синтаксис, не значення змінних під час виконання. Це пояснює, чому патч пройшов усі свої перевірки і задеплоївся чисто, а нижче порогу все одно 0.

## 3. Goal
Коли сума кошика (до знижок, в грн) нижче порогу безкоштовної доставки — поле "cost" в обох НП-квотах (відділення і адресна доставка) повинно повертати реальний порахований тариф Нової Пошти, а не 0. Вище порогу — залишити як є, вже коректно.

## 4. What to change
У `extension/PintaNovaPoshtaCod/catalog/model/shipping/pinta_nova_poshta.php`:
- Знайти в обох місцях виклику `getBoosterShippingCost(...)` (колишні `'cost' => 0,`), яка саме змінна там реально містить результат `getDocumentPrice()` на момент виклику.
- Якщо це не `$shipping_cost_default` — передавати в `getBoosterShippingCost()` правильну змінну замість неї.
- Якщо назва та сама, але значення все одно не доходить (рядок з форматуванням, 0/null саме в цей момент тощо) — знайти, чому `is_numeric($shipping_cost_default) && (float)$shipping_cost_default > 0` не проходить, і виправити це місце.
- Перед правкою підтвердити реальне ім'я і значення змінної прямо в живому файлі — не вгадувати нову назву наосліп.
