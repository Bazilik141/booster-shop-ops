**⚠️ Конфлікт isolation-патча з QA-планом:** якщо isolation-патч задеплоєний, `getMethods()` повертає `[]` безумовно — метод не з'явиться НІДЕ, включно з `checkout/checkout` за прямим URL. **Точна причина поточного бага (`Could not call registry key error!`):** OC4 `Controller::__get()` проксі лише до зареєстрованих у registry сервісів (`db`, `request`, `load`, `user`, `language` тощо). `$this->error` — НЕ вбудована властивість; кожен контролер має сам оголосити `protected array $error = [];`. Клас `MonoChast` (адмін) цього не зробив, тому `$this->error['warning'] = ...` в методі `validate()` іде через `__get()`, шукає сервіс `error`, не знаходить — падає. Ламає `index()` (збереження) і `confirmApplication()`/`rejectApplication()` (реальні виклики `/api/order/confirm` та `/api/order/reject`) — всі викликають `validate()`.

**Ключові файли (вже задеплоєні, перший патч):**
- `extension/mono_chast/catalog/model/payment/mono_chast.php` — `getMethods()`, гейт по `payment_mono_chast_status` + мін. сума + валюта UAH.

## 8. Ризики

- **`getMethods()` — спільний OC4-механізм:** у вікні, коли isolation відкочено для тесту, увімкнення `payment_mono_chast_status=1` потенційно показує метод у ЖИВОМУ SimpleCheckout, а не лише на `checkout/checkout`. Це головна причина тримати `status=0` поза вікном тесту і повертати isolation одразу після.
- **`checkout/checkout.twig` — майбутній єдиний живий чекаут (RD-13-редизайн), активно допрацьовується паралельно іншими задачами** (RD-13/ST-2c/CHECKOUT-00x серії). Будь-яка зміна тут HIGH-RISK за замовчуванням — стане живою для всіх на ST-2c cutover.
- **Isolation-патч конфліктує з sandbox-QA:** задеплоєний isolation робить `getMethods()` порожнім безумовно — метод не з'явиться навіть на `checkout/checkout` за прямим URL. Тест вимагає тимчасового відкату (розділ 5.3).
- **Координація з `ST-2c`:** cutover свідомо заблокований на PAY-001 (`blockedBy: ['PAY-001']`, коміт `4fae967`) — але подвійну перевірку лишаємо: перед cutover явно перевірити `payment_mono_chast_status=0` і стан isolation-патча; перед тестовим увімкненням mono_chast — статус ST-2c.
- Немає справжнього staff-only access-контролю для `checkout/checkout` — лише відсутність посилань з сайту (не пароль, не IP-allowlist). Тестувати коротко, вимикати `payment_mono_chast_status` одразу після.
- NCRM-міграція `0013_pay001_mono_payment_types.sql` та зміни `order-sync/index.ts` вже в git (`1b001a1`), робоча копія чиста; статус деплою міграції на Supabase не підтверджено — перевірити перед тим як покладатись на нові `payment_type_code`.
