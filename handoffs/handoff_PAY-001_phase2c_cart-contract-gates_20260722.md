## 2. Context

Рішення власника по 7 пунктах контракту (2026-07-22):

1. ✅ «Обрати» = штатний cart-add обраного товару (поточна кількість + всі опції) через існуючий ендпоінт → редірект на сток-чекаут ТІЛЬКИ після успіху cart-add. Існуючі рядки кошика не замінюються — стандартна OpenCart-семантика (merge, ліміти, помилки).
2. ✅ «Продовжити покупки» — друга кнопка в одному рядку з першою; закриває модалку, нічого не додає і не зберігає.
3. ⚠️ ЗМІНЕНО щодо контракту: прев'ю-поріг на сторінці товару рахується БЕЗ вгадування купонів (сума кошика + ціна×обрана кількість). Фінальний контроль купонів — на чекауті (розділ 4, пункт C).
4. ✅ Будь-який товар з фактичним залишком < 1 у кошику → кредит недоступний для всього замовлення. Server-side.
5. ✅ Кількість понад залишок → штатна відмова cart-add, без редіректу.
6. ✅ Скоуп: лише товарна сторінка + сток-чекаут. Категорії/пошук/картки — НЕ чіпати.
7. ⏳ Тексти кнопок і всі візуальні рішення (приглушена плашка, формат підказок, алерт) — ВИНЕСЕНІ в окремий дизайн-бриф PAY-001-UI2. Codex не вигадує фінальні тексти/стилі сам — бере з дизайн-пакета, який власник додасть до цього handoff.

**Термінологія «передзамовлення» — щоб утрьох не натупити.** Існують ДВА різні «передзамовлення»:
- **Статус ТОВАРУ** (stock status): власник ставить товару «Передзамовлення» замість «Закінчився» при нульовому залишку. Для гейтів цього патча критерій — **фактичний залишок товару < 1** (quantity), незалежно від назви stock status. Codex should verify against actual project files, чи достатньо quantity, чи треба враховувати і stock_status_id.
- **Статус ЗАМОВЛЕННЯ** «Передзамовлення» (ORDER-STATUS-001, вже в БД): власник ставить його вручну на замовлення-передзамовлення до приходу товару на склад. **У гейтах цього патча НЕ використовується** — на момент чекауту замовлення ще не існує.

## 3. Goal

Клієнт з товарної сторінки одним кліком додає товар у кошик і потрапляє на сток-чекаут з обраним строком кредиту; кредитна плашка видима на всіх товарах з чесними станами (активна / приглушена з підказкою); кредит неможливо оформити для замовлення з передзамовленням або сумою нижче порогу — і UI, і сервер це гарантують незалежно.

## 4. What to change

**A. Модалка товарної сторінки:**
- Primary-кнопка: штатний cart-add (кількість + опції) → редірект після success. Помилка cart-add = показ штатної помилки, БЕЗ редіректу.
- Secondary-кнопка «Продовжити покупки» поряд в одному рядку: закрити модалку, нічого не писати.
- Тексти обох кнопок — з дизайн-пакета PAY-001-UI2.

**B. Кредитна плашка на товарній сторінці — три стани (зміна щодо патча 20260722, який ховає блок при qty<1):**
1. Активна: залишок ≥ 1 і (кошик + ціна×кількість) ≥ поріг.
2. Приглушена «не дотягує до порогу»: залишок ≥ 1, сума нижче порогу → блок видимий, кнопка неактивна, підказка «доступно від N грн» (+ скільки лишилось). Активація без перезавантаження при зміні кількості/кошика.
3. Приглушена «передзамовлення»: залишок < 1 → блок видимий, кнопка неактивна, коротка підказка (робочий текст до дизайн-пакета: «Недоступно для передзамовлень»).

**C. Чекаут — м'який блокер порогу:**
- Якщо обрано кредит і фінальна сума до сплати (після купонів/знижок) < порогу → лаконічний friendly-алерт (сенс: «обери інший спосіб оплати або додай товарів до N грн»; фінальний текст — з дизайн-пакета) + кнопка «Підтвердити замовлення» неактивна, доки умова не усунута (зміна способу оплати або сума ≥ порогу).
- Server-side: віртуальний Mono-метод не повертається/відхиляється, якщо фінальна сума (після знижок) < порогу АБО в кошику є товар із залишком < 1. Браузерний стан — лише підказка, не істина.

**D. Копірайт/UI-фікси, підтверджені раніше:**
- Порядок чипів завжди фіксований: «3 платежі», «4 платежі», «5 платежів» — змінюється лише active-стан, не порядок.
- «5 платежів» — з буквою «в».
- Прибрати підпис «Розстрочка від підключених банків» на способі оплати (досі присутній у патчі 20260722).

## 5. Do not touch

- SimpleCheckout isolation (`PAY-001-SIMPLE-CHECKOUT-ISOLATION` у mono-моделі) і `system/library/url.php`.
- Hutko / COD / IBAN — логіка і нормалізація (регресія обов'язкова, зміни заборонені).
- Order-write boundary (ST-2b6e/2b6d гейти), checkout/confirm створення замовлення.
- Mono API payload, confirm/pending bridge, callback/poll (`extension/mono_chast/.../mono_chast.php` — поза скоупом 2c).
- Фіскалізація/Checkbox, NCRM order-sync, DB-схема і settings.
- sitemap.xml, robots.txt, redirects, canonical, .htaccess, Merchant feed, schema.
- Статус замовлення «Передзамовлення» (ORDER-STATUS-001) — не підключати до гейтів.

## 7. Acceptance criteria

1. Товар 800 грн, кількість 1: primary-кнопка модалки → товар у кошику (1 шт, опції збережені) → редірект на `?route=checkout/checkout`, обраний строк збережений, кредит преселектнутий.
2. «Продовжити покупки» → модалка закрита, кошик/сесія без змін (перевірка: лічильник кошика не змінився).
3. Товар 200 грн, кількість 1, порожній кошик → плашка видима, приглушена, підказка з сумою; зміна кількості на 3 → кнопка активується без перезавантаження.
4. Товар із залишком 0 → плашка видима, приглушена, підказка про передзамовлення; кнопка неактивна.
5. Кошик 600 грн + купон −150 → на чекауті з обраним кредитом: алерт видимий, «Підтвердити замовлення» disabled; зняття купона або вибір COD → кнопка активна.
6. Кошик містить товар із залишком 0 → у сток-чекауті віртуальний Mono-метод відсутній у відповіді payment_method (перевірка JSON), незалежно від суми.
7. Чипи завжди в порядку 3/4/5; текст «5 платежів»; підпис «Розстрочка від підключених банків» відсутній.
8. Кількість понад залишок → штатна OpenCart-помилка, без редіректу.
9. SimpleCheckout (інкогніто, status=1 у тест-вікні): mono_chast відсутній.
10. Регресія: Hutko/COD/IBAN обираються і зберігаються на сток-чекауті як до патча.
