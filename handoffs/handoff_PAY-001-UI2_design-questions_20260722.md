# Design Brief — PAY-001-UI2: дизайн-питання кредитного флоу (для Claude Design)

Date: 2026-07-22 | Parent: `handoffs/CODEX - PAY-001-credit-flow.md` (базова специфікація, вже реалізована) + `handoffs/handoff_PAY-001_phase2c_cart-contract-gates_20260722.md` (технічний контекст)

## 1. Висновок

Базовий кредитний UI (кнопка на товарі → модалка вибору 3/4/5 → чекаут) уже живий у коді. У ході QA виникли нові UI-стани, яких нема в базовій специфікації. Потрібні дизайн-рішення по 5 блоках нижче — результат піде Кодексу разом з технічним handoff Phase 2c. Це збірний документ і для майбутніх дизайн-питань цієї задачі: нові питання додаються сюди, не розпорошуються.

## 2. Тип задачі

UX/UI специфікація (розширення існуючої). Кінцевий формат — як у `CODEX - PAY-001-credit-flow.md`: конкретні тексти, стани, розміри/відступи, мобільна поведінка.

## 3. Owner

Owner → Claude Design (напрацювання) → Codex (імплементація разом з Phase 2c).

## 4. Статус

Open — питання зібрані, дизайн-рішень ще нема.

## 5. Наступна дія

Owner проганяє цей бриф через Claude Design, зберігає результат поряд (наприклад `CODEX - PAY-001-UI2-answers.md`) і віддає Кодексу обидва файли разом з handoff Phase 2c.

## 6. Дизайн-питання

**Q1. Дві кнопки модалки (в одному рядку).**
Primary виконує ДВІ дії: додає товар у кошик і веде на чекаут. Secondary закриває модалку без дій. Потрібно: фінальні тексти обох кнопок (кандидати primary: «Обрати» / «Додати в кошик і оформити» — або кращий короткий варіант), візуальна ієрархія (primary чорна заливка, secondary outline?), поведінка на мобільному (два в рядку чи стек), стан loading у primary поки триває cart-add.

**Q2. Приглушена плашка «не дотягує до порогу» (товар з залишком, ціна нижче 500 грн).**
Блок видимий на всіх товарах. Потрібно: візуал приглушення (opacity/грейскейл/інше), текст підказки з динамічною сумою (сенс: «Оплата частинами доступна від 500 грн — додайте ще N грн»), формат підказки (рядок під плашкою / tooltip / бейдж), анімація/перехід при активації без перезавантаження (клієнт збільшив кількість — плашка «ожила»).

**Q3. Приглушена плашка «передзамовлення» (фактичний залишок < 1).**
Потрібно: коротка френдлі-фраза (робочий варіант: «Недоступно для передзамовлень» — запропонувати кращі), чи візуально відрізняється від стану Q2, чи показувати обидві причини, якщо збігаються.

**Q4. М'який алерт на чекауті (обрано кредит, сума після купона < порогу).**
Потрібно: текст (лаконічний, без канцеляриту; сенс: «сума замовлення нижче 500 грн — обери інший спосіб оплати або додай товарів») і вигляд (колір/іконка/розміщення біля способу оплати чи біля кнопки), стан неактивної кнопки «Підтвердити замовлення» (сірий + курсор?), як алерт зникає при усуненні умови.

**Q5. Стани чипів 3/4/5 (закріплення).**
Порядок завжди 3 → 4 → 5, активний лише підсвічується. Підтвердити фінальний вигляд active/inactive/hover з існуючої специфікації — Кодексу потрібен один канонічний опис.

## 7. QA checklist

- Всі тексти українською, без канцеляриту і КАПСУ (крім існуючого бейджа «СКОРО БУДЕ»).
- Мобільний рендер 375px для кожного стану.
- Жодних нових обіцянок/умов кредиту в текстах (умови визначає банк) — не вигадувати відсотки/строки.
- Стилі в межах існуючої дизайн-системи boostershop-ds (кольори/радіуси/типографіка як у базовій специфікації).

## 8. Ризики

- **ПУМБ-брендинг:** будь-які зміни візуалів з логотипом/назвою ПУМБ не фіналізувати без погодження макетів банком (договірна вимога) — для цього брифа ПУМБ-картку не чіпати, лишити як є.
- Тексти підказок не повинні звучати як відмова магазину («не можна») — тон: підказка, як зробити кредит доступним.
- Розсинхрон з Codex: якщо дизайн змінить структуру DOM плашки — Кодекс має отримати обидва документи одночасно, не окремо.
