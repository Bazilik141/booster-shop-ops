### 6.2 Контекст
Власник хоче підключити «Покупку Частинами» (ПЧ) monobank (юридичний емітент — АТ «Універсал Банк») як новий спосіб оплати: безвідсоткова для клієнта розстрочка на 3–25 платежів, комісію платить магазин. **Реальна тарифна сітка підтверджена 2026-07-19** (офіційна сторінка `monobank.ua/chast/vendors`, скріншот власника): 3 платежі/2 міс → 2.9%, 4/3міс → 4.1%, 5/4міс → 5.9%, 6/5міс → 7.2%, 7/6міс → 8.3%, 8/7міс → 9.5% (сітка триває вище 8, не знадобиться — бізнес-ліміт власника 5 платежів). Для наших 3 значень (3/4/5) комісія вже не приклад, а підтверджені числа.

На сайті вже є `catalog/controller/payment/hutko.php` (картка через Hutko) і COD з фіскалізацією через Checkbox (див. handoffs ST-2b.6 / 2b.6b / 2b.6c / 2b.6d) — це способи оплати ВСЕРЕДИНІ звичайного чекауту. На сторінці товару (RD-10, `plans/RD-redesign-roadmap-plan_2026-05-30.md`, `plans/audit_2026-06-11.md`) вже є текстовий плейсхолдер **«Оплата частинами ПУМБ»**.

**Архітектурне уточнення власника (2026-07-18):** «Покупка в кредит» — ОКРЕМИЙ вид покупки, не спосіб оплати всередині звичайного чекауту. monobank ПЧ і ПУМБ мають існувати ПАРАЛЕЛЬНО як дві опції в цьому окремому потоці. Референс — власник надав скріншоти COMFY.ua: на товарній сторінці під основною кнопкою «Купити» стоїть окремий блок/кнопка «Купуй щастинами або в кредит» з прев'ю «від N ₴ в місяць»; клік відкриває модалку «Виберіть кредитну пропозицію» зі списком карток-провайдерів (лого, к-сть платежів «від 3 до 15», сума на місяць, бейдж «Без першого платежу») — ПриватБанк, ПУМБ, «Плати Щастинами без карти», Sense bank, Alliance та інші. Це UI-патерн для Phase 2, не Phase 1 (нижче).

Отже: плейсхолдер ПУМБ НЕ замінюється — він стає однією карткою в спільній модалці поряд з новою карткою monobank ПЧ. Codex не чіпає існуючу ПУМБ-логіку/бекенд, тільки спільну розмітку блоку/модалки, куди додається нова картка.

### 6.5 Що змінити (Phase 1)
- Новий payment-метод — **окремий OC4-extension за зразком `extension/hutko`** (виправлено 2026-07-19: Codex звірив з реальним бекапом — Hutko НЕ лежить у `catalog/controller/payment/`, це повноцінний OC4-extension; попередня версія цього handoff містила неперевірене припущення з тексту старого хендофа, воно було неточним). PAY-001 будується паралельним extension, той самий структурний патерн, що й Hutko — не в `catalog/controller/payment`.
- Розрахунок підпису — PHP з офіційної докс (розділ 7.A нижче), використати як основу.
- Виклики API: `POST /api/order/create` (з `result_callback` на наш новий endpoint), `POST /api/order/confirm`, `POST /api/order/reject`, `POST /api/order/state` (fallback-опитування, якщо callback не прийшов протягом розумного часу).
- Новий callback-контролер: перевіряє заголовок `signature` тим самим HMAC-методом, відкидає при неспівпадінні (401/403); читає `order_id`/`state`/`order_sub_state`; логує перехід.
- Мапінг статусів monobank → OpenCart order status (розділ 7.D): Codex перевіряє реальні `order_status_id` в адмінці, **не вигадує номери**. Мінімум: `WAITING_FOR_STORE_CONFIRM` (ключовий — можна видавати товар, після видачі викликати `/api/order/confirm`), `SUCCESS/ACTIVE`, `SUCCESS/DONE`, `SUCCESS/RETURNED`, будь-який `FAIL/*`.
- Зберігання `store-id`/`store_secret`: той самий механізм, що і для ключів Hutko (Codex перевіряє де саме, повторює патерн). **Не хардкодити, не логувати відкритим текстом.**
- Ідемпотентність: унікальний `store_order_id` на замовлення; дублікат запиту з тим самим `store_order_id` повертає наявний `order_id`, не створює новий (задокументовано офіційно).
- Retry: лише 5xx з експоненційним backoff; 4xx (валідаційні) — без повтору.
- Логування всіх переходів статусу (знадобиться `Trace-Id` з відповідей API для звернень у підтримку monobank).
- **Бізнес-правило — мінімальна сума:** спосіб ПЧ доступний лише при `total_sum` ≥ 500 грн (сума кошика/замовлення, без якого показ кнопки/пункту оплати взагалі не рендериться — перевірка і на бекенді перед `order/create`, не тільки візуально).
- **Бізнес-правило — без націнки:** `total_sum`, що йде в `order/create`, дорівнює звичайній ціні товару (+ доставка, якщо застосовно) без додавання комісії монобанку — комісію покриває магазин окремо (розрахунки з монобанком, не з ціни для клієнта).
- **Бізнес-правило — ліміт платежів:** власник хоче максимум 5 платежів. Якщо API не дає параметра для цього (розділ 6.4, технічний нюанс) — реалізувати як маркетингову рамку в нашому UI/копірайті, а не як технічний ліміт запиту, і чітко повідомити власнику, що це не гарантія на боці монобанку.

### 6.9 DB-схема Phase 1 — запит Codex на явне підтвердження (2026-07-19)

Codex звірив реальний бекап і просить explicit owner approval перед написанням коду (за AGENTS.md: DB-зміни лише з явним дозволом власника + rollback SQL у шапці патчу). Три пропозиції:

1. **Нова таблиця `ocp5_mono_chast_transaction`** — idempotency (`store_order_id` ↔ monobank `order_id`), стан callback/polling, `Trace-Id`. **Моя оцінка: однозначно потрібна, без цього немає ні ідемпотентності, ні можливості діагностувати збої через підтримку монобанку (Trace-Id). Рекомендую підтвердити як є.**
2. **5 нових OpenCart order-статусів** (очікує клієнта / очікує видачу / активна / завершена-повернена / відхилена). Контекст: у живому бекапі зараз **лише один статус в реальному використанні — «1: В обробці»**, тому мапінг «іменами» без нових статусів неможливий (Codex правий). **Дві речі варто вирішити свідомо, не мовчки:**
   - Офіційна API-довідка монобанку має **окремий ендпоінт `/api/order/return`** (розділ 6.10 нижче) — тобто «повернення» це окрема дія банку, не просто фінальний стан. Об'єднання `SUCCESS/DONE` і `SUCCESS/RETURNED` в один OC-статус «завершена/повернена» ризикує сплутати звичайне закриття ПЧ з поверненням у CRM/NCRM (де повернення тягне за собою COGS-реверс, а завершена оплата — ні). **Пропоную тримати їх окремими статусами** (6 замість 5), якщо Codex не бачить іншої причини їх зливати.
   - **Перевірено напряму в репо (не гіпотеза):** NCRM (нова Supabase CRM, `ncrm/supabase/migrations/`) має власну таблицю `order_statuses` із фіксованими кодами — рівно 7: `new, processing, shipped, received, cancelled, refund, preorder` (`0002_stage2_sales.sql:517-525`). Sync-функція з OpenCart (`0011_opencart_order_sync.sql:113-115`, дзеркально в `0012_...partial_insert.sql:161-163`) шукає `order_status_code` з вхідного payload і **тихо підставляє `'new'`, якщо код не знайдено** (`coalesce(nullif(...), 'new')`) — без помилки, без попередження. Це означає: якщо додати нові OC-статуси на кшталт «Очікує клієнта»/«Активна» без відповідного коду в NCRM, монобанк-замовлення просто осядуть як `new` у новій CRM, а це прямо заважає NCRM-10 (order pipeline OpenCart→Supabase, зараз In progress).
   - **Пропозиція на розгляд** (не остаточна — Codex перевіряє з повним доступом до OC-хуків): замість 5 абсолютно нових статусів спробувати класти проміжні стани ПЧ (`WAITING_FOR_CLIENT`/`WAITING_FOR_STORE_CONFIRM`) в існуючий OC-статус «В обробці» (він і так єдиний активний зараз, точний стан веде `ocp5_mono_chast_transaction`), а фінальні — мапити на вже існуючі NCRM-коди де це чесно підходить: `ACTIVE`/видача товару → `shipped`, `RETURNED` → вже існуючий `refund`, `FAIL/*` → `cancelled`. Це закриває NCRM-10 без нової міграції там. Якщо власник/Codex вважають, що потрібна більша гранулярність саме в OC — тоді питання нижче про синхронну міграцію NCRM `order_statuses` стає обов'язковим кроком того ж патчу, не окремим TODO «колись».
3. **Конфіг extension у `ocp5_setting` / `ocp5_extension`** — стандартний патерн реєстрації розширення. **Без зауважень, рекомендую підтвердити.**

### 6.11 Відкрите питання «чи можна обмежити до 5 платежів» (розділ 6.4 п.6) — ЗАКРИТО 2026-07-19

Власник надав повний приклад payload з офіційної докс (JS-розгорнута версія, недоступна простому fetch). **Підтверджено остаточно:** так, технічно можливо й саме так це і працює. Повна структура запиту `/api/order/create`:

```json
{
  "store_order_id": "ORD-2024-001234",
  "client_phone": "+380501234567",
  "total_sum": 2499.99,
  "invoice": {
    "number": "INV-2024-001234",
    "date": "2024-01-15",
    "source": "INTERNET",
    "point_id": "STORE-001"
  },
  "available_programs": [
    { "type": "payment_installments", "available_parts_count": [3, 6, 10] }
  ],
  "products": [
    { "name": "Телевізор Samsung 55\"", "count": 2, "sum": 9999.99 }
  ],
  "result_callback": "https://shop.example.com/api/order/callback",
  "additional_params": { "seller_phone": "+380501234568", "nds": 416.67, "ext_initial_sum": 500 },
  "financial_company_merchant_info": { "store_name": "...", "edrpou_code": "...", "iban_account": "..." }
}
```
Відповідь `201`: `{"order_id": "fa4a8249-336e-4e6d-9b85-79bc8be62377"}`.

**Для ліміту «максимум 5»:** магазин сам передає `available_programs[0].available_parts_count` — масив дозволених кількостей платежів (у прикладі монобанку `[3, 6, 10]`). Booster Shop надсилає `[3, 4, 5]` (або підмножину) — і клієнт у застосунку монобанку фізично не побачить варіантів понад це. Ліміт реалізується технічним параметром запиту, не лише маркетинговою рамкою в копірайті. Це закриває питання остаточно — **Codex може закладати `available_parts_count` як конфігурований масив (дефолт `[3,4,5]`), а не вигадану відсутню можливість.**

Додатково видно з прикладу:
- `invoice` потребує реальних `number`/`date`/`source`(`"INTERNET"` для нас)/`point_id` (видає монобанк при реєстрації партнером).
- `products` — по одному об'єкту на товар кошика: `name`, `count`, `sum`.
- `additional_params.nds` — сума ПДВ. **Уточнити перед Phase 2: чи Booster Shop платник ПДВ** (репо натякає на спрощену систему — `Taxer_import_*.csv`, `payment_types.fop_control` у NCRM); якщо ні — імовірно 0 або поле не передається, не вигадувати без підтвердження власника/бухгалтера.
- `financial_company_merchant_info` — optional, ймовірно не потрібен для нашої моделі партнерства (звичайний магазин, не фінансова компанія), але звірити при реєстрації.

### 6.14 NCRM: order_status / payment_status / payment_type для кредитних замовлень — рішення власника + технічний дизайн (2026-07-19)

**Рішення власника (спрощує розділ 6.9 п.2 вище):** нові кредитні замовлення (mono, ПУМБ, інші банки згодом) в NCRM одразу отримують order_status **«Нове»**, payment_status **«не оплачено»**, і новий payment_type **«кредит [Банк]»**. Далі власник вручну, за звичайною процедурою, переводить payment_status в «оплачено», коли кошти від банку надходять. Жодної нової автоматизації для цього фліпу не потрібно.

**Наслідок для розділу 6.9 п.2:** це **знімає ризик з order_status** — жодного нового OC/NCRM order_status не треба, `order_status_code = 'new'` це саме той код, що вже існує в NCRM (`0002_stage2_sales.sql:519`) і саме той, куди тихо падає sync при невідомому коді (розділ 6.9), тож тут навіть немає різниці між «свідомо» і «за замовчуванням». П.1 (нова transaction-таблиця) і п.3 (extension config) лишаються як є.

**Але виникає новий, окремий технічний момент — `payment_type`, не піднятий Codex:** NCRM-таблиця `payment_types` (`0002_stage2_sales.sql:18-28, 484-507`) — це НЕ проста мітка, кожен код прив'язаний до `fee_pct_config_key`/`fee_fixed_config_key`/`fee_min_config_key` для розрахунку комісії в фінзвітності NCRM. Зараз лише 4 коди: `fop_control`, `cod_personal`, `bank_details`, `acquiring` (дефолтний fallback при невідомому коді — розділ 6.9). Комісія монобанку (2,9%+ залежно від к-сті платежів) технічно й економічно НЕ те саме, що комісія еквайрингу (`acquiring`) — якщо просто підставити код `acquiring` для кредитних замовлень, NCRM порахує неправильну маржу/P&L по цих замовленнях.

**Пропоную (технічний дизайн, «повна свобода» від власника):**
- Нова NCRM-міграція: `payment_types` отримує окремі коди на банк×к-сть платежів — `credit_mono_3`/`credit_mono_4`/`credit_mono_5` («Кредит Моно 3/4/5»), `credit_pumb_*` на майбутнє, з власними `fee_pct_config_key` під реальну комісійну сітку монобанку (окрема від acquiring-конфіга). **Реальні значення підтверджено 2026-07-19:** `credit_mono_3` → 2.9%, `credit_mono_4` → 4.1%, `credit_mono_5` → 5.9% (джерело — офіційна сторінка `monobank.ua/chast/vendors`, розділ 6.2).
- OpenCart-хук/sync payload для кредитних замовлень передає `payment_type_code = 'credit_mono'` (не `acquiring`), `order_status_code = 'new'`, `payment_status_code = 'unpaid'`.
- Ця NCRM-міграція логічно йде **в парі** з OC-стороною PAY-001 Phase 1 (той самий раунд підтвердження, не окремий TODO) — інакше перші ж кредитні замовлення або впадуть у `acquiring` з неправильною комісією, або (якщо код зовсім не передати) в дефолтний fallback.

**Важливий нюанс по таймінгу «не оплачено → оплачено»:** офіційний опис сервісу монобанку прямо каже — «магазини отримують оплату миттєво» (не після завершення всіх платежів клієнта). Тобто фактичне надходження коштів на рахунок Booster Shop, ймовірно, відбувається одразу/наступного банківського дня після підтвердження ПЧ клієнтом, а не розтягнуто на весь термін розстрочки. Це не змінює запропонований власником процес (все одно ручний фліп «за звичайною процедурою»), але варто звірити реальний таймінг виплат при реєстрації партнером — якщо це справді Т+0/Т+1, ручний фліп можна робити майже одразу після підтвердження, а не «колись пізніше». **Уточнено розділом 6.15 нижче: «миттєво» відраховується від виклику `/api/order/confirm`, тобто від фактичної відправки товару, не від підтвердження ПЧ клієнтом.** Монобанк також згадує окремий ендпоінт `/api/store/report` («Денний звіт операцій та фінансова аналітика») — потенційно придатний пізніше для звірки виплат, поза скоупом Phase 1, деталі — розділ 6.15.

### 6.15 Повна API-довідка «Заявки на ПЧ» — усі 9 файлів прочитано (2026-07-19)

Власник надав рештy офіційної докс — усі ендпоінти, яких бракувало в розділі 6.10 (там був тільки `/api/order/create`). Повні схеми запит/відповідь — розділи 7.I–7.Q нижче. Тут — лише висновки, що впливають на дизайн Phase 1 і на процес власника:

- **`/api/order/confirm` (7.J) підтверджує пункт 8 розділу 0:** перший платіж списується з клієнта тільки в момент виклику `confirm` магазином, не раніше і не автоматично після `WAITING_FOR_STORE_CONFIRM`. Дизайн «викликати confirm після реальної видачі товару» (розділ 6.5) — вже правильний, тепер це підтверджено докс, а не припущення.
- **`/api/order/return` (7.L) підтверджує пункт 2 розділу 0:** повернення — окрема дія з власним ключем ідемпотентності `store_return_id`, підтримує часткові повернення, а повне повернення вимагає передати ПОВНУ початкову суму (не залишок). «6 статусів, не 5» (DONE окремо від RETURNED) — тепер факт з докс, не рекомендація з обережності.
- **`/api/order/check/paid` (7.N) і `/api/store/report` (7.O) — різні речі, не плутати:** `check/paid` перевіряє, чи КЛІЄНТ повністю розрахувався за розстрочку (`fully_paid`, відповідає статусу `SUCCESS/DONE`) — це не про те, чи магазин отримав гроші. `/api/store/report` — денний звіт саме про виплати МАГАЗИНУ: по кожному замовленню `commission_percent`, `commission`, `transferred_sum` (скільки реально зайшло після комісії), `pay_parts`. Це готовий інструмент для ручної звірки «не оплачено→оплачено» з розділу 0 п.7 — не автоматизація (власник просив без неї), а звіт, який можна відкрити вручну замість пошуку по банківській виписці. Поза скоупом Phase 1, але власнику варто знати, що цей звіт існує.
- **`/api/store/report` також може бути реальним джерелом відсотків комісії по 3/4/5 платежах** (поле `commission_percent` в кожному рядку звіту) — придатний, якщо колись знадобиться звірити цифри для `app_config` з розділу 0 п.6 / розділу 6.14 замість (або на додачу до) `monobank.ua/chast/vendors`.
- **`/api/v2/client/validate` (7.Q):** перевірка, чи телефон належить клієнту monobank, ДО повного `order/create`. Можливе Phase 2 UX-покращення (не питати номер телефону наосліп) — але в докс явно є `429` (rate limit) серед відповідей, тож використовувати економно (напр. лише перед відправкою форми, не на кожен ввід цифри).
- **`/api/order/data` (7.M) і `/api/order/guarantee/letter` (7.P):** довідкові/сервісні ендпоінти (детальна інформація про замовлення з історією повернень; PDF гарантійного листа). Не блокують Phase 1, знадобляться для підтримки клієнтів пізніше — не додаю в критерії приймання Phase 1.

## 7. Довідка — API monobank «Покупка Частинами» (витяг з наданої докс, для самодостатності хендофа)

### 7.A Автентифікація
Кожен запит — заголовки `store-id` (ідентифікатор магазину) і `signature`.
`signature = Base64(HMAC-SHA256(request_body_bytes, store_secret))`

```php
function calculateSignature($storeSecret, $requestBody) {
    return base64_encode(hash_hmac('sha256', $requestBody, $storeSecret, true));
}
// приклад тіла: {"store_order_id":"ORD-001","client_phone":"+380501234567","total_sum":1500.00}
```

### 7.B Флоу інтеграції
1. Клієнт обирає товари й оплату через ПЧ.
2. Магазин: `POST /api/order/create` (опційно `result_callback`).
3. Клієнт отримує push у застосунку monobank, підтверджує договір ПЧ.
4. Статус → `IN_PROCESS/WAITING_FOR_STORE_CONFIRM`.
5. Якщо задано `result_callback` — monobank сам POST-ить на нього; інакше — магазин опитує `POST /api/order/state`.
6. Магазин видає товар → `POST /api/order/confirm` (або `POST /api/order/reject` при відмові у видачі).

### 7.C Callback
Надсилається **лише у 2 випадках**: перехід у `IN_PROCESS/WAITING_FOR_STORE_CONFIRM`, і перехід у будь-який `FAIL/*`. Для інших станів (напр. `IN_PROCESS/WAITING_FOR_CLIENT`) callback НЕ надсилається — обов'язковий polling через `/api/order/state`.

Формат:
```json
{"order_id": "fa4a8249-336e-4e6d-9b85-79bc8be62377", "state": "IN_PROCESS", "order_sub_state": "WAITING_FOR_STORE_CONFIRM"}
```
Безпека: перерахувати підпис тим самим методом, порівняти із заголовком `signature` callback-запиту, обробляти лише при збігу.

### 7.D Довідник статусів
| Статус | Підстатус | Опис | Дія |
|---|---|---|---|
| SUCCESS | ACTIVE | Товар видано, ПЧ активна | Фінальний, дій не треба |
| SUCCESS | DONE | ПЧ повністю сплачена | Фінальний |
| SUCCESS | RETURNED | Товар повернено, гроші клієнту | Фінальний |
| IN_PROCESS | WAITING_FOR_CLIENT | Очікує підтвердження клієнта в застосунку | Чекати, callback не приходить — polling |
| IN_PROCESS | **WAITING_FOR_STORE_CONFIRM** | Клієнт підтвердив — можна видавати товар | Видати товар + `/api/order/confirm` |
| FAIL | CLIENT_NOT_FOUND | Не клієнт monobank / не знайдено | Перевірити формат телефону |
| FAIL | EXCEEDED_SUM_LIMIT | Перевищено ліміт ПЧ клієнта | Зменшити суму / інший спосіб оплати |
| FAIL | EXISTS_OTHER_OPEN_ORDER | У клієнта вже є відкрите замовлення ПЧ | Почекати 15 хв або скасувати старе |
| FAIL | NOT_ENOUGH_MONEY_FOR_INIT_DEBIT | Не вистачає коштів на перший платіж | Клієнт поповнює картку |
| FAIL | REJECTED_BY_CLIENT | Клієнт відхилив ПЧ | Інший спосіб оплати |
| FAIL | PAY_PARTS_ARE_NOT_ACCEPTABLE | Неприйнятна к-сть платежів з боку магазину | Звернутись у підтримку monobank |
| FAIL | FRAUD_REJECTED | Відхилено антифрод-системою | Звернутись у підтримку |
| FAIL | RESTRICTED_BY_RISKS | Обмеження ризик-менеджменту | Звернутись у підтримку |
| FAIL | CLIENT_PUSH_TIMEOUT | Клієнт не відповів за 15 хв | Повторити пізніше |
| FAIL | REJECTED_BY_STORE | Магазин сам скасував | Фінальний |
| FAIL | FAIL | Внутрішня помилка банку | Повторити через 5 хв |

### 7.F Обробка помилок
HTTP: `200` успіх · `201` замовлення створено · `400` bad request (валідація) · `401` підпис відсутній/невалідний · `403` заборонено · `404` не знайдено · `429` rate limit · `500` внутрішня помилка.
Формат помилки: `{"message": "текст помилки"}`.

### 7.G Best practices (з докс)
Валідувати дані клієнта до відправки; унікальний `store_order_id`; завжди вказувати `result_callback`; обробляти ідемпотентність; негайно обробляти `WAITING_FOR_STORE_CONFIRM`; перевіряти підпис callback; мати fallback-polling; логувати всі зміни статусу; повторювати лише 5xx з експоненційним backoff; при зверненні в підтримку — додавати `Trace-Id`, Store ID, Order ID, приклади запитів/відповідей, timestamps.

### 7.I `POST /api/order/state` — перевірка статусу заявки (fallback-polling)
Request: `{"order_id": "fa4a8249-336e-4e6d-9b85-79bc8be62377"}`
Response `200`: `{"order_id": "...", "state": "IN_PROCESS", "order_sub_state": "WAITING_FOR_STORE_CONFIRM", "message": "..."}` — той самий формат, що й callback (розділ 7.C), значення `state`/`order_sub_state` — за довідником 7.D.

### 7.J `POST /api/order/confirm` — підтвердження видачі товару
Request: **лише** `{"order_id": "..."}` — жодних додаткових полів.
Response `200`: `{"order_id", "state", "order_sub_state", "message"}`.
Ефект: статус → `SUCCESS/ACTIVE`, і **саме в цей момент з клієнта списується перший платіж** — не раніше, не автоматично після `WAITING_FOR_STORE_CONFIRM`. Тому виклик має бути прив'язаний до реальної відправки/видачі товару (розділ 6.5, 6.15, розділ 0 п.8).

### 7.K `POST /api/order/reject` — скасування заявки магазином
Request: `{"order_id": "..."}`
Response `200`: той самий формат `order_id`/`state`/`order_sub_state`/`message`.
Ефект: статус → `FAIL/REJECTED_BY_STORE`, кредитний договір анулюється, клієнта сповіщає банк, кредитний ліміт клієнта відновлюється. Викликати лише ДО видачі товару (немає в наявності, технічна проблема тощо) — не після `confirm`.

### 7.L `POST /api/order/return` — повернення товару (повне або часткове)
Request: `{"order_id": "...", "sum": 2499.99, "store_return_id": "RET-2024-001", "return_money_to_card": true, "additional_params": {"nds": 416.67}}`
- `sum` — обов'язково, ≥ 0.01; для ПОВНОГО повернення передається ПОВНА початкова сума замовлення, не залишок.
- `store_return_id` — обов'язково, власний ключ ідемпотентності для повернень (окремий від `store_order_id`).
- `return_money_to_card` — boolean: миттєве повернення на картку клієнта чи повернення готівкою в магазині.
Response `200`: `{"status": "OK"}`.
Умови: замовлення має бути в статусі `SUCCESS`; сума не може перевищувати непогашений залишок; можливі кілька часткових повернень поспіль; повне повернення переводить у `SUCCESS/RETURNED` (підтверджує розділ 0 п.2 і 6.9 п.2 — це самостійна дія, не те саме, що `DONE`).

### 7.M `POST /api/order/data` — детальна інформація про заявку
Request: `{"order_id": "..."}`
Response `200`: `{"total_sum", "source", "invoice_number", "invoice_date", "point_id", "store_order_id", "create_timestamp", "reverse_list": [{"sum", "timestamp"}], "maskedCard", "iban"}` — включно з історією повернень (`reverse_list`). Довідковий/підтримка-ендпоінт, не в критеріях приймання Phase 1.

### 7.N `POST /api/order/check/paid` — чи клієнт повністю розрахувався
Request: `{"order_id": "..."}`
Response `200`: `{"fully_paid": true, "bank_can_return_money_to_card": true}`.
**Не плутати з 7.O:** це про клієнта (відповідає `SUCCESS/DONE`), не про надходження коштів магазину.

### 7.O `POST /api/store/report` — денний звіт операцій
Request: `{"date": "2024-01-15"}`
Response `200`: `{"orders": [{"order_id", "invoice_number", "order_date", "pay_parts", "commission_percent", "total_sum", "transferred_sum", "commission", "operation_timestamp", "odb_contract_number"}]}`.
Готовий звіт саме про виплати магазину (`transferred_sum` — скільки зайшло після комісії, `commission_percent`/`commission` — реальні цифри по кожному замовленню, `pay_parts` — фактична к-сть платежів). Кандидат на ручний інструмент звірки «не оплачено→оплачено» (розділ 0 п.7, розділ 6.15) — не автоматизація, власник відкриває вручну за потреби.

### 7.P `POST /api/order/guarantee/letter` — гарантійний лист
Request: `{"order_id": "...", "invoice": {"number": "...", "date": "..."}}`
Response `200`: PDF-файл (бінарний, без JSON-схеми в докс); помилки — стандартний формат `{"message": "..."}`. Сервісний ендпоінт для підтримки клієнтів, не в критеріях приймання Phase 1.

### 7.Q `POST /api/v2/client/validate` — валідація клієнта за телефоном
Request: `{"phone": "+380501234567"}`
Response `200`: `{"found": true}`. Помилки: `400/401/403/429/500` (є rate limit — `429`). Можливе Phase 2 UX-покращення (перевірити номер до повного `order/create`), використовувати економно через ліміт запитів. Застарілий v1-варіант також існує в докс — використовувати v2.

### 7.R Бренд/naming (з файлу «Про бренд»)
Дозволені написання: «Покупка Частинами», «Покупка Частинами monobank», «Покупка Частинами monobank | Universal Bank». **Не писати:** «Оплата частинами monobank Монобанк», «АТ Universal Bank», «Кредит monobank». Позиціонування — «безкоштовна розстрочка», обов'язково комунікувати клієнту, що сервіс надається за рахунок магазину. Емітент: АТ «Універсал Банк», ліцензія НБУ №92 від 20.01.1994, № 226 у держреєстрі банків, учасник Фонду гарантування вкладів. Підключення: Telegram-бот `@chast_monobankbot` → «Стати партнером» → анкета → API (потрібен програміст) або через бот (покрокова інструкція) → ~3 дні. **Повна тарифна сітка підтверджена 2026-07-19** (`monobank.ua/chast/vendors`, скріншот власника): 3 платежі → 2.9%, 4 → 4.1%, 5 → 5.9%, 6 → 7.2%, 7 → 8.3%, 8 → 9.5%.

## 🐞 Update 2026-07-21 (раунд 8) — крашиться збереження налаштувань, знайдено причину в коді

Owner спробував зберегти sandbox-креденшли → фатальна помилка: `Could not call registry key error!` (`system/engine/controller.php:43`), бектрейс веде в `extension/mono_chast/admin/controller/payment/mono_chast.php:20` (метод `validate()`), викликаний з `index()` на рядку 8.

**Причина підтверджена напряму з коду першого патча (не гіпотеза):** метод `validate()` пише `$this->error['warning'] = ...` і читає `!$this->error`, але клас `MonoChast` НЕ оголошує власну властивість `protected array $error = [];`. У OC4 `Controller::__get()` — це проксі саме до зареєстрованих сервісів (registry: `db`, `request`, `load`, `user`, `language` тощо), а не загальна властивість для помилок; коли `$this->error` не оголошено як реальна властивість класу, PHP іде через магічний `__get()`, той шукає `'error'` у registry, не знаходить — звідси помилка. Це не проблема БД/конфігу, суто відсутнє оголошення властивості.

**Наслідок ширший, ніж просто збереження налаштувань:** `confirmApplication()`/`rejectApplication()` (реальні виклики `/api/order/confirm` та `/api/order/reject`) ТЕЖ викликають `validate()` першим рядком — тобто зараз повністю непрацездатна вся адмінська частина екстеншена, не тільки форма налаштувань.

**Фікс для Кодекса (простий, точковий):** додати `protected array $error = [];` у тіло класу `Opencart\Admin\Controller\Extension\MonoChast\Payment\MonoChast` в `extension/mono_chast/admin/controller/payment/mono_chast.php`. Файл вже задеплоєний на сервері (не через інсталятор-патч, самі файли з першого раунду) — Кодекс готує окремий точковий патч (правка одного файлу, без зачеплення DB/registry/settings).

**Owner:** креденшли ще НЕ збереглися (форма впала до записи) — sandbox поки не налаштований, тестування 4 телефонів досі не почалось.

**Окремо, не блокує цей фікс:** щодо «новий чекаут лише для власника, клієнти на старому» — я не звіряв це в коді (`getMethods()` — це модель, спільна для будь-якого checkout-контролера, що її викликає); варто самому перевірити інкогніто/на старому чекауті перед тим як покладатись на це як на гарантію ізоляції, коли дійде до реального ввімкнення методу.
